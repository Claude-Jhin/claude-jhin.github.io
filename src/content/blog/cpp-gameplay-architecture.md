我以前写玩法逻辑的时候，最容易掉进去的坑就是：
**功能刚开始很好写，但一旦需求叠起来，类会迅速变得又胖又难改。**

这几年我越来越倾向于把“玩法代码是否舒服”归结为两件事：

- 状态切分是否清晰
- 数据和行为是否被合理拆开

## 少一点“万能 Actor”

在原型阶段，把所有逻辑都塞进一个 Actor 或 Component 里很常见。
但项目一旦进入持续迭代，这种写法的成本会突然变高。

我现在更喜欢让一个类只承担一个明确责任，比如：

- 输入解释
- 状态推进
- 表现触发
- 数值配置读取

这样做的好处不是“优雅”，而是后面加需求时不会牵一发而动全身。

## 我喜欢的一个拆法

例如角色技能系统，我常常会先把行为拆成这三层：

```cpp
struct FSkillConfig
{
    float Cooldown = 0.0f;
    float CastTime = 0.0f;
    int32 ManaCost = 0;
};

class USkillRuntimeState
{
public:
    bool bIsCasting = false;
    float RemainingCooldown = 0.0f;
};
```

配置和运行时状态分开以后，很多东西都会自然起来：

- 数值策划更容易接入
- 调试时能快速判断是配置错还是状态错
- 回放、存档、网络同步也更容易想清楚

## 数据驱动不只是为了策划

很多人说数据驱动，第一反应是“方便策划配表”。
但对程序自己来说，数据驱动的最大好处其实是让代码结构更稳。

当规则变化频繁时，把变化点挪到数据里，代码就能更多承担“解释规则”的角色，而不是“硬编码每条规则”。

## 最后

我越来越不追求那种一开始看起来很炫的架构了。
对我来说，真正好的玩法代码是：

- 三周后自己回来还能迅速读懂
- 新需求插进来时不用大面积返工
- 出 bug 时能快速缩小范围

这听起来朴素，但在游戏开发里往往非常值钱。
