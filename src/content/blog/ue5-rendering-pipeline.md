最近我在回头整理自己对 UE5 渲染流程的理解，发现很多时候我们说“画面出问题了”，其实是在不同阶段看到了不同的症状。

真正有帮助的方式不是死记整条渲染管线，而是先把几个最常排查的节点记清楚：

1. `Scene` 数据是怎么被组织和提交的
2. `Base Pass` 和 `Lighting` 是怎么累积出主要观感的
3. `Post Process` 又是如何把最终气质彻底改掉的

## 我会先看哪几层

当一个画面“看起来不对”时，我通常会先问自己：

- 是材质响应不对？
- 是光照结果不对？
- 还是后处理把一切都推偏了？

这三类问题看起来都像“画面脏了”或者“层次没了”，但根源完全不同。

## Base Pass 不是“只是把颜色画上去”

刚开始接触延迟渲染时，很容易把 Base Pass 理解成“先把东西画出来的一步”。
但在实际调试里，它更像是把后面大量光照计算所需的信息，先完整写进 GBuffer。

我现在更习惯把它理解成：

- 不是在“完成画面”
- 而是在“准备画面被正确点亮的前提”

一旦法线、粗糙度、金属度或者材质表达错了，后面所有光照都会跟着偏。

## Lumen 带来的调试心态变化

UE5 里引入 Lumen 之后，我的调试心态有一个挺明显的变化：
以前我会更快地怀疑灯光摆放，现在我会先怀疑“间接光是否正在把体积关系重新定义”。

尤其在室内场景里，下面几种情况很常见：

- 直射光没问题，但空间还是发灰
- 阴影边缘不脏，却缺少层次
- 材质参数正确，但整体显得“发空”

很多时候并不是贴图不行，而是间接光能量、表面响应和曝光一起把空间压平了。

## 我希望自己记住的观察顺序

为了避免每次都被细节拖走，我给自己定了一个简单顺序：

```cpp
// 先判断问题属于哪一层，再决定进哪个工具看
enum class VisualIssueLayer
{
    Material,
    Lighting,
    PostProcess
};
```

虽然这只是一个抽象分类，但它确实能让我在分析时快很多。

## 最后

我越来越觉得，图形调试最重要的不是“知道多少术语”，而是能不能快速把错误归类。
当你知道自己是在看材质、光照还是后处理时，很多问题都会突然变得简单。
