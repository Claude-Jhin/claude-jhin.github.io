最近遇到一个有点绕的网络问题：平时 Perforce 和 P4V 都能正常使用，但一旦开启某个 VPN 工具，命令行和图形界面就开始表现异常。

这篇笔记把当时的排查过程整理下来，方便后面遇到类似问题时少走一点弯路。文中的主机名、IP 和端口都做了脱敏处理，用统一的占位符表示。

现象

问题主要表现为两种：

  1. 命令行执行 p4 info 失败
  2. P4V 在开启 VPN 后无法正常连接，或者连上后很快断开

排查过程中看到过两类典型报错：

  • WSAEACCES
  • WSAECONNRESET

前者更像本地网络访问被拦截，后者更像连接建立后又被远端或中间链路强制断开。

第一步：先确认端口到底通不通

最先做的是直接看 Perforce 服务端口是否可达：

这里真正值得看的，不只是 TcpTestSucceeded，还包括 SourceAddress。

对比开启和关闭 VPN 时的结果，可以发现：

  • 开启 VPN 时，SourceAddress 显示为 VPN出口地址_A
  • 关闭 VPN 时，SourceAddress 显示为 本地网卡地址_A

这一步说明了一件很关键的事：Perforce 流量在两种状态下，实际走的是两条完全不同的出网路径。

第二步：对比 VPN 开关前后的 DNS 解析结果

接着对 Perforce 的主机名做了解析测试：

结果非常明确：

  • 关闭 VPN 时，P4_HOST 解析到 P4内网地址_A
  • 开启 VPN 时，P4_HOST 解析到 P4公网地址_A

这一步基本就把问题框住了。问题不只是“流量要不要经过 VPN”，而是 VPN 直接改写了 DNS 解析结果。

也就是说，开启 VPN 之后，Perforce 访问的已经不是原本那条可用的内网路径，而是被导向了另一条公网路径。

第三步：结合 P4V 日志确认行为差异

再去看 P4V 日志，现象会更清楚一些。

关闭 VPN 时，这些请求都能正常执行：

  • info
  • protects
  • client -o
  • stream -o
  • changes

这说明几件事情都没有问题：

  • P4V 本身配置没问题
  • 账号和票据没问题
  • Perforce 服务本身也没问题

开启 VPN 后，请求一开始往往能发出去，但随后会出现：

  • TCP receive failed
  • WSAECONNRESET

这就意味着真正的问题不是 Perforce 自己坏了，而是会话在网络层被切到了另一条不稳定或不正确的链路上，最终被重置。

根因分析

把上面的现象串起来，问题本质就比较清楚了：

  1. VPN 改写了 P4_HOST 的 DNS 解析结果
  2. 同时也改变了目标连接的流量出口
  3. Perforce 因此访问到了错误的地址，或者走上了错误的网络路径
  4. 最终导致 P4V 和 p4 表现为连接异常、连接重置或访问被拒绝

所以这次问题的根因并不在 Perforce 本身，也不是 P4V 配错了，而是 VPN 介入后改变了“解析到哪里”和“流量从哪里出去”这两件事。

结论

这次故障最核心的经验，不是“Perforce 出问题了”，而是“VPN 改变了解析和出网路径，导致 Perforce 被带到了错误的目标地址上”。

如果以后再遇到类似问题，我会优先检查这三件事：

  1. 用 nslookup 看主机名在不同网络环境下到底解析到哪里
  2. 用 Test-NetConnection 看 SourceAddress，确认流量到底从哪里出去
  3. 对比 P4V 日志，判断是本地拦截、链路重置，还是单纯的端口不可达

一句话总结就是：Perforce 连不上，不一定是服务挂了，也可能只是 VPN 把它带到了错误的 DNS 结果和错误的网络出口上。
