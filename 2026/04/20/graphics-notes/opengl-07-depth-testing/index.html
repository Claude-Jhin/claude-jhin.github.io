在进入高级 OpenGL 之后，第一件真正影响 3D 真实感的事情，其实不是更复杂的光照，也不是更炫的特效，而是“前后的遮挡关系到底对不对”。

如果没有深度测试，OpenGL 更像是在做一件很朴素的事：谁后画，谁就把前面画过的内容盖掉。
这在 2D 场景里不一定是问题，但在 3D 里几乎立刻就会出错。

这一节的核心目标，就是建立一个非常稳定的直觉：

  • 颜色缓冲决定“像素是什么颜色”
  • 深度缓冲决定“哪个像素应该留在最前面”

一、这节到底解决了什么问题

假设场景里有两个立方体和一块地板：

  • 立方体离摄像机更近
  • 地板离摄像机更远

如果只按绘制顺序来处理，而不比较前后关系，那么只要地板最后画，它就会错误地盖到前面的立方体上。
这不是模型数据错了，也不是矩阵错了，而是 GPU 根本不知道“谁更靠近观察者”。

深度测试就是为了解决这件事。

它让 GPU 在写入像素颜色之前，先比较当前片段和深度缓冲中的深度值，然后决定：

  • 当前片段应该通过
  • 还是应该被丢弃

所以这节最重要的一句话是：

深度测试解决的不是“画不画”，而是“谁应该显示在前面”。

二、深度缓冲里存的是什么

深度缓冲可以先粗略理解成一张“只存距离信息的图”。

屏幕上的每个像素位置，除了有颜色值，还会额外对应一个深度值。
这个深度值不是世界坐标里的原始 z，而是经过投影和后续变换后，用来参与深度比较的结果。

你现在先不用急着把所有数学细节一次吃完，只要先记住：

  • 每个片段都会带一个深度值
  • 深度缓冲会记录当前这个像素位置上“已经通过测试的最近深度”
  • 新来的片段要先和这个值比较

如果更近，就替换；如果更远，就丢弃。

三、开启深度测试的最关键两句

这一节代码里最核心的设置是：

它们的含义分别是：

glEnable(GL_DEPTH_TEST)

开启深度测试功能。

如果没有这句，OpenGL 在默认情况下不会替你处理 3D 场景里的前后遮挡关系，结果更接近“按绘制顺序覆盖”。

glDepthFunc(GL_LESS)

设置深度测试的比较规则。

GL_LESS 的意思是：

  • 如果当前片段的深度值比深度缓冲里已有的值更小
  • 那么当前片段通过测试

这通常就可以理解成：

  • 更靠近摄像机的片段保留
  • 更远的片段被挡住

这也是最常用、最符合直觉的一种深度比较方式。

四、为什么每帧都要清深度缓冲

只开深度测试还不够，还必须记住这句：

很多初学者刚进这节时，会觉得：

  • 我已经清了颜色缓冲
  • 为什么还要清深度缓冲

原因很直接：

  • 颜色缓冲存的是上一帧画出来的颜色
  • 深度缓冲存的是上一帧留下来的深度结果

如果你不清深度缓冲，那么新一帧的片段在做深度比较时，会和“上一帧残留的深度数据”比，结果就会莫名其妙。

所以一旦使用深度测试，就要把这条规则记牢：

每一帧通常都要同时清颜色缓冲和深度缓冲。

五、我这次实验里最值得观察的现象

为了把这节讲得更直观，我把项目写成了一个对比实验。

场景里有：

  • 两个立方体
  • 一块地板
  • 摄像机可移动
  • T 键可切换深度测试开关

并且我故意让地板在渲染顺序上“最后绘制”。

这样就会出现两种非常清楚的结果：

当深度测试开启

  • 更近的立方体会正确挡住地板
  • 即使地板最后画，遮挡关系仍然是对的

当深度测试关闭

  • OpenGL 不再关心谁更近
  • 只关心谁最后画
  • 因为地板最后画，所以它会错误地覆盖本该挡住它的立方体部分

这个对比特别好，因为它会让你从“记 API”变成“理解现象”。

六、这一节和前面内容是什么关系

深度测试不是孤立出现的，它是建立在你前面已经学会的 3D 渲染链路之上的。

你前面已经有了：

  • 顶点数据
  • model / view / projection
  • 摄像机移动
  • 纹理采样

这意味着你已经能把 3D 场景画出来了。

而这一节做的事情是：

  • 不再只问“能不能画出来”
  • 开始问“画出来之后，前后关系对不对”

所以从学习节奏上说，这一节是很自然的一步。
它把“3D 看起来像真的”这件事推进了一大截。

七、这一节代码里最该抓住的几处

如果只从代码层面抓重点，我会优先记这几件事：

1. 深度测试状态可以动态开关

项目里用了一个布尔值来控制当前是否启用深度测试，并把它绑定到 T 键上。

这件事很适合学习，因为你能立刻看到同一个场景在两种状态下的区别。

2. glEnable(GL_DEPTH_TEST) 和 glDisable(GL_DEPTH_TEST)

这两个 API 让你看到，深度测试本质上也是 OpenGL 的一项状态。

3. glClear(GL_DEPTH_BUFFER_BIT)

它提醒你：深度缓存不是抽象概念，而是每帧都要被维护的真实缓冲区。

4. 渲染顺序仍然存在，但它不再是唯一决定因素

这点特别重要。

开启深度测试后，渲染顺序并没有消失。
只是最终哪个片段能留下，不再完全由“谁最后画”决定，而是由深度比较参与共同决定。

八、这一节最容易混淆的几个点

1. 深度测试不是“只给 3D 模型用的高级特效”

它其实是 3D 渲染里非常基础的一层机制。
只要场景里存在前后遮挡，它几乎就是默认需要的。

2. 深度缓冲和颜色缓冲不是一回事

  • 颜色缓冲：存颜色
  • 深度缓冲：存深度比较结果

它们通常要一起工作，但职责完全不同。

3. 开启深度测试以后，不代表就可以完全不管绘制顺序

对于不透明物体，深度测试通常能帮你把大部分遮挡关系处理正确。
但到了后面的混合、透明物体阶段，绘制顺序又会重新变得很重要。

所以这节不是终点，而是给后面的“模板测试、混合、面剔除”打基础。

九、这一节我现在应该能回答的问题

学完这节之后，至少应该能自己回答下面这些问题：

  1. 深度测试解决的是什么问题？
  2. 为什么 3D 场景不能只靠绘制顺序？
  3. glEnable(GL_DEPTH_TEST) 是在做什么？
  4. glDepthFunc(GL_LESS) 的含义是什么？
  5. 为什么每帧要清 GL_DEPTH_BUFFER_BIT？
  6. 为什么关闭深度测试后，后绘制的地板会错误覆盖前面的立方体？

如果这六个问题你都能比较顺地答出来，这一节就算真正进脑子了。

十、这一节的收获

这一节最值得带走的，不是某一个 API 名字，而是一种新的观察角度：

在 3D 渲染里，一个像素最终能不能显示在屏幕最前面，除了颜色，还要经过一轮“前后关系”的判断。

而这层判断，就是深度测试。

所以这一节可以收成一句总结：

颜色告诉 GPU“画成什么样”，深度测试告诉 GPU“谁应该留在最前面”。

后续阅读会继续沿着这个方向展开：

  • 深度测试解决“前后”
  • 模板测试开始解决“哪些区域允许被画、哪些区域不允许被画”

这条高级 OpenGL 的主线，到这里就真正开始展开了。
