在刚开始学 OpenGL 的时候，我对 Shader 的理解其实非常抽象。

我知道它是“运行在 GPU 上的小程序”，也知道顶点着色器和片元着色器是现代 OpenGL 的核心，但如果让我回答一个更具体的问题，比如：

“顶点颜色是怎么从 C++ 代码一路传到屏幕上的？”

或者：

“为什么一个 uniform 变量就能让整个三角形动起来？”

那时其实并没有真正想清楚。

这一篇文章记录的，就是我继续沿着 LearnOpenGL 学到“着色器”这一节时，逐渐把这些问题串起来的过程。

对应的本地项目是：

  • projects/learnopengl/learnopengl/04_shaders

这一次我不再把 Shader 源码直接写死在 .cpp 字符串里，而是把它们拆到了独立文件中，并且第一次在顶点数据里加入了“颜色属性”，再通过 uniform 做了一个最小的动态效果。

如果说上一篇文章解决的是“OpenGL 程序怎样跑起来”，那么这一篇更像是在回答：

“OpenGL 程序真正画图时，CPU 和 GPU 到底是怎么配合的？”

为什么着色器这一节很关键

OpenGL 的前两步，更多是在建立外壳：

  • 创建窗口
  • 创建上下文
  • 加载 OpenGL 函数
  • 搭建渲染循环

而到了 Shader 这一节，程序才第一次真正开始触碰“图形内容本身”。

因为从这里开始，画面上的变化不再只是来自固定颜色清屏，而是来自 GPU 上运行的程序。

这意味着：

  • 顶点放在哪里，不再是 OpenGL 帮你默认决定的
  • 像素是什么颜色，也不再是 OpenGL 自动给的
  • 这些事情都需要你明确地写在 Shader 里

从这个角度看，Shader 不是 OpenGL 的某个附属知识点，它就是现代 OpenGL 的核心。

这一节到底学了什么

在这次实践里，我重点做了 4 件事：

  1. 把顶点着色器和片元着色器拆到外部文件里
  2. 写了一个最小的 Shader 类来统一读取、编译和链接 Shader
  3. 让每个顶点同时携带“位置 + 颜色”两类属性
  4. 用 uniform 从 CPU 向 GPU 传一个每帧变化的偏移量

看起来只是几步小改动，但它们其实刚好覆盖了着色器这一节最重要的几个核心概念：

  • Shader 是怎样组织的
  • 顶点属性是怎样送进顶点着色器的
  • 顶点着色器和片元着色器是怎样通过 out/in 协作的
  • uniform 又是怎样从 CPU 传给 GPU 的

先从 Shader 文件本身开始看

这次我没有把 Shader 继续写在 C++ 的长字符串里，而是拆成了两个文件：

  • shaders/shader.vs
  • shaders/shader.fs

首先，它说明这个顶点着色器接收两份输入：

  • aPos：顶点位置
  • aColor：顶点颜色

其次，它说明这个着色器会把颜色通过 out vec3 ourColor 传给后面的片元着色器。

最后，它还声明了一个 uniform float xOffset，这个值不是来自某个具体顶点，而是来自 CPU 端在每一帧统一设置的全局输入。

在 main() 里，顶点着色器做了两件事：

第一句表示：

  • 把顶点位置写到 gl_Position
  • 并在 x 方向额外加上一个偏移量

第二句表示：

  • 把当前顶点携带的颜色传递给后面的片元阶段

也就是说，顶点着色器在这里负责两件事：

  1. 决定顶点放到哪里
  2. 把颜色这个属性继续往后传

片元着色器在做什么

片元着色器比顶点着色器更短：

它做的事情非常直接：

  • 接收从顶点着色器传过来的颜色
  • 把这个颜色写成最终像素颜色

这里有一个非常值得注意的点：

片元着色器拿到的 ourColor 并不是简单照搬某个顶点的颜色，而是经过了图元插值之后的结果。

因为三角形有 3 个顶点，每个顶点颜色都不一样，所以在三角形内部的每个片元上，颜色会平滑过渡。

这也是为什么最终看到的不是一个纯色三角形，而是带有渐变效果的三角形。

C++ 代码是怎么把 Shader 用起来的

这一节里，我第一次正式写了一个自己的 Shader 类。

对应文件是：

  • Shader.h

这个类做的事情，本质上就是把下面这几步打包起来：

  1. 从文件读取 Shader 源码
  2. 创建顶点着色器对象和片元着色器对象
  3. 编译两个 Shader
  4. 链接成一个可用的 Program
  5. 提供一些方便设置 uniform 的辅助函数

为什么要封装一个 Shader 类

如果不做封装，每次使用 Shader 都要重复写：

  • 打开文件
  • 读取源码
  • glCreateShader
  • glShaderSource
  • glCompileShader
  • 检查编译错误
  • glCreateProgram
  • glAttachShader
  • glLinkProgram
  • 检查链接错误

这些流程本身没有问题，但它们都属于很标准、很重复的样板代码。

所以把它们放进 Shader 类里，带来的好处很明显：

  1. 主程序逻辑更清晰
  2. Shader 的读取、编译、链接过程集中管理
  3. 后面增加更多 Shader 文件时更容易复用

构造函数做了什么

Shader 类的构造函数做的事情可以概括成一句话：

“从两个文件路径出发，构造出一个可用的 Shader Program。”

它先用 ifstream 把 shader.vs 和 shader.fs 读成字符串，然后分别创建顶点着色器和片元着色器对象，再完成编译和链接。

如果中间哪一步失败，就会把驱动返回的错误日志打印出来。

这一点非常重要，因为 Shader 出问题时，最常见的表现就是“窗口正常、程序正常、但画面什么都没有”。

这时候如果没有编译和链接日志，排错会非常痛苦。

顶点数据为什么从 3 个 float 变成了 6 个 float

上一篇里，我画三角形时每个顶点只包含位置：

  • x
  • y
  • z

而在这一节里，每个顶点变成了：

  • x
  • y
  • z
  • r
  • g
  • b

也就是说，每个顶点现在包含两类属性：

  1. 位置
  2. 颜色

这一步的意义很大，因为它让我第一次真正意识到：

“顶点并不只是位置，它可以携带很多不同类型的信息。”

后面继续学下去，顶点里还可能有：

  • 法线
  • 纹理坐标
  • 切线
  • 骨骼权重

而这一节中的颜色属性，就是最简单的一个入门版本。

glVertexAttribPointer(...) 终于不再只是背 API 了

uniform 到底是什么

uniform 是这一节里另一个非常关键的概念。

在当前这个例子里，顶点着色器声明了：

这意味着：

  • CPU 每帧根据时间计算一个偏移量
  • 再把这个值传给 GPU
  • GPU 里的顶点着色器用这个值去修改顶点位置

所以这个三角形看起来会左右轻微摆动。

这里我觉得最值得记住的一点是：

uniform 不是“每个顶点各有一份的数据”，而是“当前这次绘制统一使用的一份数据”。

换句话说，它更像是这次 draw call 的全局参数。

这和顶点属性有本质区别：

  • 顶点属性：每个顶点都有自己的值
  • uniform：整次绘制共享同一个值

把这两类输入区分开，是我在这一节里最大的收获之一。

这一页真正把哪条链路打通了

如果把这次的程序逻辑压缩成一句话，那么我现在对它的理解是：

  • C++ 代码准备了带位置和颜色的顶点数据
  • 通过 VBO 把这些数据上传到 GPU
  • 通过 VAO 告诉 OpenGL 如何解释这些数据
  • 通过 Shader Program 让 GPU 知道顶点该怎么处理、颜色该怎么输出
  • 通过 uniform 把每一帧的偏移量传给顶点着色器
  • 最后通过 glDrawArrays 发出绘制命令

这一节最容易混淆的几个点

1. 顶点属性和 uniform 的区别

  • 顶点属性：每个顶点一份
  • uniform：整次绘制共享一份

2. out 和 in 的关系

  • 顶点着色器的 out 会传给片元着色器的 in
  • 中间会经过插值，而不是简单地原样拷贝某一个顶点的值

3. Shader 类并没有改变 OpenGL 原理，只是把样板代码收起来了

封装之后代码更清楚了，但底层依然还是：

  • 读文件
  • 编译 Shader
  • 链接 Program
  • 设置 uniform

到目前为止，我对 Shader 的理解发生了什么变化

学这一节之前，我对 Shader 的印象更像是：

“OpenGL 里必须有的某种特殊代码。”

学完这一节之后，我开始更具体地理解它：

  • 顶点着色器负责处理顶点输入，并决定顶点最终位置
  • 片元着色器负责决定屏幕上每个像素的颜色
  • Shader Program 是把这两者连接起来的一套完整 GPU 程序
  • uniform 是 CPU 和 GPU 之间非常重要的一种参数传递方式
  • 顶点属性则是另一条输入通道，它们是按顶点逐个送入顶点着色器的

换句话说，这一节让我第一次不只是“会用 Shader”，而是开始知道 Shader 代码为什么要这么写。

这一节收尾时，我确认自己已经掌握的东西

截至现在，我希望自己至少已经真正掌握了下面这些点：

  1. 知道如何从文件读取并构建一个 Shader Program
  2. 知道 layout(location = N) 必须和 glVertexAttribPointer(N, ...) 对上
  3. 知道一个顶点可以同时携带多种属性
  4. 知道 out/in 会在 Shader 阶段之间传递数据
  5. 知道 uniform 是 CPU 发给 GPU 的全局输入
  6. 知道真正触发绘制的依然是 draw call，而不是前面那些准备动作

下一步

继续往下学时，我会特别关注两件事：

  1. Shader 如何继续和纹理、变换矩阵这些内容结合起来
  2. 顶点属性在后面的纹理坐标、法线等场景里会怎样继续扩展

因为我已经能感觉到，后面的很多内容其实都只是这一节的自然延伸：

  • 颜色属性会变成纹理坐标、法线等更多属性
  • uniform float 会变成矩阵、向量、材质参数等更多输入
  • 片元着色器也会从“输出固定颜色”逐渐进化成“做真正的光照计算”

结语

如果上一篇文章让我第一次理解了 OpenGL 程序是怎么“跑起来”的，那么这一篇更像是让我第一次看清：

OpenGL 程序到底是怎么“开始真正画东西”的。

而 Shader、顶点属性和 uniform，正是这条链路里最重要的三个入口。
