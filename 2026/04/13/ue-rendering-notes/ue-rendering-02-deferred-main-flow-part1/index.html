写在前面

上一篇笔记整理的是渲染正式开始之前的准备工作，核心是 ViewFamily -> FSceneView -> FViewInfo 这条链。
这一次开始，关注点从“视图如何被准备好”切换到“这一帧到底如何被组织起来并执行”。

本文覆盖的范围，是主场景 Deferred Renderer 从 RenderThread 入口开始，到：

  • PrePass
  • Velocity
  • BasePass
  • Deferred Lighting
  • Fog / Cloud / Water / Translucency

这一大段为止。

对应地，本文暂时还不展开：

  • PostProcessing 的内部链路
  • Tonemap / Bloom / MotionBlur 的细节
  • 更细的 shader 级实现

这一篇最核心的目标，是先把整帧主骨架建立起来。

这一步最重要的理解是：

  • RenderViewFamily_RenderThread 是 RenderThread 侧统一的帧入口
  • FDeferredShadingSceneRenderer::Render 是 Deferred Renderer 的整帧总调度入口

前者更像执行外壳，后者才是真正组织这一帧的地方。

2. Render() 一开始为什么不是 PrePass

第一次点进 Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:1736 时，很容易期待立刻看到 RenderPrePass 或 RenderBasePass。实际并不是这样。

Render() 一开头先做的是：

  • 判定 RendererOutput
  • 判定 bNaniteEnabled
  • 判定 bHasRayTracedOverlay
  • 初始化 ray tracing scene view state
  • 调用 OnRenderBegin(...)
  • 启动曝光 LUT、VT、Light Function Atlas 等系统更新
  • 提交 CommitFinalPipelineState()
  • 准备 Lumen、SkyAtmosphere、Nanite 的本帧状态

最关键的几个位置是：

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:1749
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:1751
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:1790
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:1813
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:1837

这段代码最准确的理解不是“杂项初始化”，而是：

在真正进入几何阶段之前，Deferred Renderer 先把这一帧的运行条件、资源依赖和关键路径开关固定下来。

3. OnRenderBegin(...)：真正的帧开始预处理

最值得单独拎出来的入口是：

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:1790
  • Engine/Source/Runtime/Renderer/Private/SceneRendering.cpp:3913

OnRenderBegin(...) 做的事情非常集中：

  • 处理挂起的 VT / 材质缓存回调
  • 组织 Scene 更新阶段允许的异步操作
  • 准备 ViewRect 和 SceneTexturesConfig
  • 调用 ViewExtension->PreRenderViewFamily_RenderThread(...)
  • 调用 ViewExtension->PreRenderView_RenderThread(...)
  • 启动可见性任务 LaunchVisibilityTasks(...)

这说明在 RenderThread 上，真正“开始这一帧”并不等于“立刻画图”，而是：

先把 Scene、View、Visibility 和 SceneTextures 的运行条件准备到位，再进入后面的几何和光照阶段。

4. 关键布尔量：整帧结构往往先由它们决定

这一段学习中最容易被忽略，但其实最关键的，是一批阶段开关。

bNeedsPrePass

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:2284
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:474

它决定这帧是否真的执行 PrePass。

bRenderDeferredLighting

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:2208

它决定是否进入 deferred lighting 主路径。

bNaniteEnabled

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:1751
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:482

它决定某些几何阶段是否走 Nanite 路径。

这一段最值得建立的意识是：

在 UE 的整帧渲染里，很多阶段不是“总会跑”，而是先经过一轮路径判定之后才真正进入执行。

5. PrePass + Velocity：真正落地的第一个几何阶段

这一段的入口在：

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:2369

对应的主体逻辑是：

  • 清理深度/模板
  • 运行 RenderPrePass(...)
  • 在某些模式下运行 RenderVelocities(...)
  • 如启用 Nanite，继续走 Nanite 几何相关路径
  • 组织 SceneDepth、PartialDepth
  • 让 SceneTextures 至少处于 SceneDepth 可用状态

真正的 PrePass 实现入口在：

  • Engine/Source/Runtime/Renderer/Private/DepthRendering.cpp:525

PrePass 到底建立了什么

它最直接的输出包括：

  • SceneDepth
  • 可选的 PartialDepth
  • 为后续 HZB、Occlusion、BasePass 深度测试建立基础

其中最值得注意的是 second stage depth pass：

  • Engine/Source/Runtime/Renderer/Private/DepthRendering.cpp:637
  • Engine/Source/Runtime/Renderer/Private/DepthRendering.cpp:646
  • Engine/Source/Runtime/Renderer/Private/DepthRendering.cpp:649

这说明 PrePass 不是总是“画一遍深度就结束”，而是有时会把第一阶段深度保留下来，再做第二阶段细化。

Velocity 到底是什么

这里的 velocity 更准确地说不是物理速度，而是 motion vector，运动向量。

它表示的是：

当前帧某个像素，相对上一帧在屏幕空间移动了多少。

所以它不是“物体每秒移动几米”，而是“这个像素上一帧大概在哪里”。

ShouldRenderVelocities() 的判断逻辑在：

  • Engine/Source/Runtime/Renderer/Private/VelocityRendering.cpp:387

它之所以重要，是因为后面很多系统都依赖它：

  • TAA
  • TSR
  • Motion Blur
  • SSR 的时序部分
  • SSGI
  • Lumen
  • Distortion

也就是说，Velocity 是现代时序渲染链里的基础资源，不只是给 motion blur 准备的附加纹理。

为什么 PrePass 和 Velocity 会挨在一起

因为它们都属于“几何前段的基础资源建立”：

  • PrePass 建立深度域基础资源
  • Velocity 建立时序和运动信息

它们都不是最终颜色输出，但都决定后续 BasePass、Lighting 和时序系统能否稳定工作。

6. BasePass：不是“再画一遍”，而是建立 opaque 场景表达

BasePass 的调用点在：

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:2905

实现入口在：

  • Engine/Source/Runtime/Renderer/Private/BasePassRendering.cpp:1071

主体在：

  • Engine/Source/Runtime/Renderer/Private/BasePassRendering.cpp:1449

BasePass 到底做了什么

最准确的定义是：

BasePass 是 opaque 场景的第一次完整材质着色阶段。

它不是纯深度阶段，也不是完整 lighting 阶段，而是在做这些事：

  • 绑定 SceneColor + GBuffer 多个 MRT
  • 根据材质和 GBuffer layout 写入材质属性
  • 建立 opaque scene color 的底板
  • 消费 DBuffer、反射、雾、EyeAdaptation 等输入

其中最关键的代码是：

  • Engine/Source/Runtime/Renderer/Private/BasePassRendering.cpp:1150

这说明 BasePass 不是单 render target pass，而是一个 MRT 阶段。

BasePass 写的是什么

从源码组织可以确定它至少在建立两类核心结果：

SceneColor

BasePass 会真正开始建立当前帧的主场景颜色缓冲。

GBuffer

BasePass 还会把后续光照求解需要的材质属性写入 GBuffer，例如：

  • 法线
  • 粗糙度
  • 金属度
  • shading model
  • base color

所以 BasePass 最核心的作用不是“把最终画面画出来”，而是：

把 opaque scene 变成后续 Deferred Lighting 能理解和消费的场景表达。

BasePass 是 GPU 阶段吗

是。

更准确地说：

  • 你读到的 RenderBasePass(...) 是 RenderThread 上的 CPU 侧组织代码
  • 但它组织出的 BasePass 是 GPU 上真实执行的 raster pass

这一点在源码里非常直接：

  • Engine/Source/Runtime/Renderer/Private/BasePassRendering.cpp:1702
  • Engine/Source/Runtime/Renderer/Private/BasePassRendering.cpp:1705
  • Engine/Source/Runtime/Renderer/Private/BasePassRendering.cpp:1709

这里明确是一个 ERDGPassFlags::Raster 的 pass，并最终通过 Pass->Draw(...) 提交绘制。

对普通 mesh 而言，BasePass 本质上就是：

基于三角形光栅化的场景绘制阶段。

7. CPU 是怎么把绘制命令发给 GPU 的

这也是这段学习里最值得澄清的一个问题。

在 UE 里，CPU 往 GPU 发命令，并不会写成一个很直白的 SubmitToGPU()。
它通常被拆成三层：

  1. RenderThread 上通过 RDG 组织 pass
  2. 在 pass 的 lambda 里用 FRHICommandList 录制 draw / dispatch
  3. 最后由 GraphBuilder.Execute() 统一执行整张图

在 PrePass 里最接近“发命令”的地方

  • Engine/Source/Runtime/Renderer/Private/DepthRendering.cpp:610
  • Engine/Source/Runtime/Renderer/Private/DepthRendering.cpp:614
  • Engine/Source/Runtime/Renderer/Private/DepthRendering.cpp:617

在 BasePass 里最接近“发命令”的地方

  • Engine/Source/Runtime/Renderer/Private/BasePassRendering.cpp:1702
  • Engine/Source/Runtime/Renderer/Private/BasePassRendering.cpp:1706
  • Engine/Source/Runtime/Renderer/Private/BasePassRendering.cpp:1709

也就是说，真正的关键位置是：

  • GraphBuilder.AddPass(...)
  • lambda 里的 Pass->Draw(...)

在 RDG 视角里，这些 pass 会先被加入一张渲染图，然后等：

  • Engine/Source/Runtime/Renderer/Private/SceneRenderBuilder.cpp:915

GraphBuilder.Execute() 被调用时，再统一调度执行。

所以更准确的理解不是“命令先攒成一个数组再统一发”，而是：

命令和资源依赖一起被组织成一张渲染图，最后按依赖顺序统一执行。

8. Deferred Lighting：不是单一 Pass，而是一整段 lighting 阶段组

Deferred Lighting 主入口在：

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3254

这段最重要的几个调用是：

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3265 RenderDiffuseIndirectAndAmbientOcclusion(...)
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3314 RenderLights(...)
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3329 第二次 RenderDiffuseIndirectAndAmbientOcclusion(...)
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3339 RenderDeferredReflectionsAndSkyLighting(...)
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3346 AddSubsurfacePass(...)

Deferred Lighting 到底在做什么

最准确的理解是：

它不是重新画几何，而是读取 BasePass 建好的 SceneColor + GBuffer + Depth，继续做光照求解与合成。

这里要特别注意一个概念：

  • BasePass 更像一个相对集中的主 pass
  • Deferred Lighting 更像一个 lighting pass group

所以从源码级别看，Deferred Lighting 并不适合简单理解成“和 BasePass 一样的一个 pass”，更准确地说它是一整段光照阶段。

它主要在读什么

  • SceneColor
  • GBuffer
  • SceneDepth
  • LightingChannelsTexture
  • SortedLightSet
  • AO / GI / Reflection / SkyLighting 等辅助结果

为什么它必须在 BasePass 之后

因为没有 BasePass，后面就没有：

  • 材质法线
  • 粗糙度
  • 金属度
  • shading model
  • scene color 底板

Deferred Lighting 本质上就是建立在这些输入之上的。

9. Fog、Cloud、Water、Translucency：后半程环境与合成阶段

Lighting 之后，整帧进入后半段环境与合成区。

最关键的入口在：

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3380
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3431
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3608

为什么它们总在 Lighting 之后

因为这几类效果大多依赖：

  • 已经形成的 opaque scene color
  • 已经建立好的 scene depth
  • 已经完成的 lighting 结果

所以它们天然不像 BasePass 那样属于“建立场景表达”的前半段，而更像：

基于已有场景结果进行体积、覆盖和混合的后半程阶段。

Fog / Cloud

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3383 ComputeVolumetricFog(...)
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3405 RenderVolumetricCloud(...)

它们不是简单画 mesh，而是：

  • 先形成体积/离屏结果
  • 再按深度和已有 scene color 合成回场景

Water

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3524 RenderSingleLayerWater(...)
  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3529

这里最值得记住的是：Single Layer Water 不只是叠一层颜色，它会改变后续阶段所使用的深度结构。

Translucency

真正大段透明主线在：

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3654

而在这之前，Renderer 已经准备了：

  • FTranslucencyPassResourcesMap
  • FSceneWithoutWaterTextures
  • translucency lighting volume
  • front layer translucency 相关资源

这说明透明阶段并不是一个“直接画完就结束”的单段 pass，而是一整组后半程合成路径的一部分。

这一阶段最值得记住的 12 个结论

  1. 主场景 RenderThread 的统一执行入口是 Engine/Source/Runtime/Renderer/Private/SceneRendering.cpp:4895 的 RenderViewFamily_RenderThread。
  2. 对 Deferred 路径来说，真正的整帧总调度入口是 Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:1736 的 FDeferredShadingSceneRenderer::Render。
  3. Render() 开头首先做的是路径判定和系统准备，而不是立即进入某个几何 pass。
  4. OnRenderBegin(...) 是 RenderThread 帧开始阶段最关键的预处理入口，定义在 Engine/Source/Runtime/Renderer/Private/SceneRendering.cpp:3913。
  5. PrePass + Velocity 是整帧第一个真正落地的几何阶段，它主要建立深度域和时序信息，而不是最终颜色。
  6. velocity 在渲染语境里更准确地说是 motion vector，也就是像素跨帧的屏幕空间运动向量，而不是简单物理速度。
  7. BasePass 是 opaque 场景的第一次完整材质着色阶段，本质上是一个基于三角形光栅化的 GPU raster pass。
  8. BasePass 的核心价值不只是写 SceneColor，更是建立 GBuffer，提供后续 Deferred Lighting 所需的材质属性。
  9. CPU 并不是在某一行“直接调用 GPU”，而是先通过 RDG 组织 pass，再在 Pass->Draw(...) / Pass->Dispatch(...) 中录制命令，最后由 GraphBuilder.Execute() 统一执行。
  10. Deferred Lighting 更准确地说是一整段 lighting pass group，而不是像 BasePass 那样单一、集中的一个 pass。
  11. Fog、Cloud、Water、Translucency 总在后半段，是因为它们大多依赖已经形成的 opaque scene color、scene depth 和 lighting 结果。
  12. 到目前为止，整帧主线已经可以稳定概括为：视图就绪 -> 深度/速度 -> opaque 场景表达 -> 光照求解 -> 环境与透明合成 -> 后处理输出。
