系列定位：UE 渲染源码主线拆解
本文主题：GameThread / RenderThread / RHIThread 的协作，以及 UPrimitiveComponent -> FPrimitiveSceneProxy 的渲染更新链路

写在前面

前面的几篇文章主要在回答两类问题：

  • 一帧画面在真正开始渲染前，是怎样被准备好的
  • Deferred Renderer 在 RenderThread 上，又是怎样把这一帧组织成 PrePass -> BasePass -> Lighting -> PostProcessing 的

但如果只停在这些阶段名上，仍然会有一个非常关键的问题没有真正讲透：

这些渲染阶段，究竟是怎样在不同线程之间协作起来的？

继续往下追源码时，几乎一定会遇到下面这些名字：

  • GameThread
  • RenderThread
  • RHIThread
  • MarkRenderStateDirty
  • SendAllEndOfFrameUpdates
  • CreateSceneProxy
  • FPrimitiveSceneProxy

如果这些概念之间的关系没有建立起来，那么后面再看 FMeshBatch、MeshDrawCommand、GPUScene、SceneVisibility，就很容易觉得“函数很多，但不知道它们属于哪一层”。

本文就专门解决这个问题。
文章会分成两部分：

  1. UE 渲染里 GameThread、RenderThread、RHIThread 到底是怎样协作的
  2. 一个 UPrimitiveComponent 的变化，是怎样一步步传到 FPrimitiveSceneProxy 和 FPrimitiveSceneInfo，最终影响画面的

1. GameThread、RenderThread、RHIThread 分别负责什么

GameThread：准备“要画什么”

GameThread 主要负责高层世界状态和视图准备。
对渲染来说，它最重要的职责不是直接画，而是先把这一帧的渲染描述整理好。

典型工作包括：

  • Actor、Component、相机和玩法状态更新
  • 构建 FSceneViewFamily
  • 创建 FSceneView
  • 生成 FViewInfo
  • 创建 FSceneRenderer
  • 把组件变化翻译成渲染系统能够消费的更新

前面已经看到，像 UGameViewportClient::Draw、BeginRenderingViewFamily、FSceneRenderer::FSceneRenderer 这些关键步骤，本质上都还处于这条线程上。

也就是说，GameThread 解决的是：

这一帧要画什么，视角是什么，哪些渲染对象发生了变化。

RenderThread：组织“怎么画”

RenderThread 真正开始执行主场景一帧的入口是：

  • Engine/Source/Runtime/Renderer/Private/SceneRendering.cpp:4895 RenderViewFamily_RenderThread

对于主场景 Deferred Renderer，真正的总调度入口是：

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:1736 FDeferredShadingSceneRenderer::Render

它负责的工作包括：

  • 消费 FSceneRenderer 和 FViewInfo
  • 组织 PrePass、BasePass、Lighting、Translucency、PostProcessing
  • 搭建 RDG 图
  • 在 pass 执行阶段录制渲染命令
  • 驱动可见性、mesh draw command、GPUScene 等渲染侧系统

所以 RenderThread 解决的是：

这一帧应该按什么阶段顺序执行，每个阶段读什么、写什么、提交什么。

RHIThread：继续向下提交

RHIThread 可以理解成更靠近底层图形 API 的那一层。
它会继续处理 FRHICommandList，把 BeginRenderPass、DrawIndexedPrimitive、DispatchComputeShader 这类命令交给 D3D12、Vulkan、Metal 等后端。

它解决的是：

这些渲染命令怎样被真正送到底层图形接口。

需要注意的是，RHIThread 是否独立存在，与平台和配置有关。
但从理解渲染架构的角度，可以先把它视为：

  • RenderThread：渲染组织层
  • RHIThread：底层提交层

2. 为什么 UE 不让 RenderThread 直接读取 Actor

这是理解线程协作的关键。

RenderThread 上当然需要知道“这个物体该怎么画”，但它不能直接去读大部分 UObject / Actor / UPrimitiveComponent 的实时状态。原因很简单：

  • 这些对象主要归 GameThread 管
  • 它们不是通用线程安全对象
  • 如果 RenderThread 直接读，会引入严重的数据竞争和线程边界混乱

所以 UE 的做法不是“跨线程共享组件本体”，而是引入一层渲染代表对象。

对 UPrimitiveComponent 来说，这层代表对象就是：

  • FPrimitiveSceneProxy

配合 scene 内部管理结构：

  • FPrimitiveSceneInfo

于是线程边界就清楚了：

  • GameThread 持有游戏对象
  • RenderThread 持有渲染对象
  • 二者通过 FScene、命令队列和 deferred update 机制协作

3. 组件变化不是立刻跨线程，而是先“打 dirty”

一个 UPrimitiveComponent 的变化，通常不会立刻把命令发给渲染线程。
UE 先做的是标记这份变化属于哪一类。

最关键的入口在：

  • Engine/Source/Runtime/Engine/Private/Components/ActorComponent.cpp:2587 DoDeferredRenderUpdates_Concurrent
  • Engine/Source/Runtime/Engine/Private/Components/ActorComponent.cpp:2634 MarkRenderStateDirty
  • Engine/Source/Runtime/Engine/Private/Components/ActorComponent.cpp:2650 MarkRenderTransformDirty
  • Engine/Source/Runtime/Engine/Private/Components/ActorComponent.cpp:2671 MarkRenderDynamicDataDirty
  • Engine/Source/Runtime/Engine/Private/Components/ActorComponent.cpp:2682 MarkForNeededEndOfFrameUpdate
  • Engine/Source/Runtime/Engine/Private/Components/ActorComponent.cpp:2719 MarkForNeededEndOfFrameRecreate

这里最重要的不是函数名本身，而是背后的分类：

MarkRenderStateDirty

表示这次变化会影响渲染状态本身，通常需要重建 render state。

MarkRenderTransformDirty

表示只是变换变了，通常只需要更新 bounds 和 local-to-world。

MarkRenderDynamicDataDirty

表示某些每帧变化的动态渲染数据需要同步，但不需要重建整个 proxy。

这套设计说明了一个很重要的事实：

UE 会先判断变化的性质，再决定要不要重建 proxy，还是只更新局部渲染数据。

4. 帧末统一冲刷：SendAllEndOfFrameUpdates

当组件把自己标记为 render dirty 后，并不是立刻就进入 RenderThread。
它通常会先进入当前 world 的“帧末待更新队列”。

统一执行入口在：

  • Engine/Source/Runtime/Engine/Private/LevelTick.cpp:1346 UWorld::SendAllEndOfFrameUpdates

Renderer 在真正开始一帧之前，会显式确保这件事先完成：

  • Engine/Source/Runtime/Renderer/Private/SceneRendering.cpp:5059

源码注释写得很清楚：

在开始 BeginRenderViewFamily 前，保证所有 render proxies 都是最新的。

这就是线程协作里的第二个核心事实：

GameThread 先把这一帧所有组件的渲染变化在帧末统一结算，再让 RenderThread 正式开始组织新的一帧。

6. 组件移动时，为什么通常不需要重建 proxy

第二种常见情况是：组件只是位置、旋转、缩放变了。

入口在：

  • Engine/Source/Runtime/Engine/Private/Components/PrimitiveComponent.cpp:657 SendRenderTransform_Concurrent

它会先更新 bounds，然后调用：

  • GetWorld()->Scene->UpdatePrimitiveTransform(this)

真正的 scene 入口在：

  • Engine/Source/Runtime/Renderer/Private/RendererScene.cpp:1556

核心逻辑在：

  • Engine/Source/Runtime/Renderer/Private/RendererScene.cpp:1566

这里会先判断：

  • 这次 transform 变化是不是普通更新
  • 还是这个组件要求“变换一更新就重新创建 proxy”
    • 对应 ShouldRecreateProxyOnUpdateTransform()

如果只是普通更新，它会组装一包：

  • WorldBounds
  • LocalBounds
  • LocalToWorld
  • AttachmentRootPosition
  • PreviousTransform

然后通过：

  • ENQUEUE_RENDER_COMMAND(UpdateTransformCommand)(...)

交给 RenderThread，见：

  • Engine/Source/Runtime/Renderer/Private/RendererScene.cpp:1635

RenderThread 侧真正处理的位置是：

  • Engine/Source/Runtime/Renderer/Private/RendererScene.cpp:1512 UpdatePrimitiveTransform_RenderThread

这说明：

大多数“组件移动了”的情况，不会重建 proxy，而是更新 proxy 对应的变换与 bounds 数据。

7. MarkRenderStateDirty 的真实代价是什么

MarkRenderStateDirty() 看起来只是一句标记，但它的真实含义通常是：

这一份渲染代表对象已经不够用了，需要整套重新来。

在帧末执行 DoDeferredRenderUpdates_Concurrent() 时，如果发现：

  • bRenderStateDirty == true

就会走：

  • RecreateRenderState_Concurrent()

这通常意味着：

  1. DestroyRenderState_Concurrent()
  2. 再 CreateRenderState_Concurrent()

对 UPrimitiveComponent 来说：

  • DestroyRenderState_Concurrent() 会调用 World->Scene->RemovePrimitive(this)
    • Engine/Source/Runtime/Engine/Private/Components/PrimitiveComponent.cpp:763
  • CreateRenderState_Concurrent() 会重新 AddPrimitive(this)
    • Engine/Source/Runtime/Engine/Private/Components/PrimitiveComponent.cpp:631

所以：

MarkRenderStateDirty 不是一个轻量标记，它常常意味着 proxy 和 scene info 的重建。

这也是为什么 UE 还要专门区分：

  • MarkRenderTransformDirty
  • MarkRenderDynamicDataDirty

因为不是所有变化都值得走最贵的那条路径。

8. 组件被移除时，又发生了什么

当组件注销或离开场景时，UPrimitiveComponent 会走：

  • DestroyRenderState_Concurrent()

见：

  • Engine/Source/Runtime/Engine/Private/Components/PrimitiveComponent.cpp:763

这里会调用：

  • World->Scene->RemovePrimitive(this)

FScene::RemovePrimitive 的入口在：

  • Engine/Source/Runtime/Renderer/Private/RendererScene.cpp:1987

内部关键逻辑在：

  • 拿到 PrimitiveSceneProxy
  • 先把组件和 proxy 解绑
  • ENQUEUE_RENDER_COMMAND(FRemovePrimitiveCommand)(...)

RenderThread 再继续执行：

  • RemovePrimitiveSceneInfo_RenderThread(...)
  • PrimitiveSceneProxy->DestroyRenderThreadResources()

见：

  • Engine/Source/Runtime/Renderer/Private/RendererScene.cpp:2040

这说明删除过程和添加过程是对称的：

  • GameThread 负责高层解绑和命令排队
  • RenderThread 真正清理渲染侧资源

9. FPrimitiveSceneProxy 为什么是线程边界的核心

FPrimitiveSceneProxy 的定义在：

  • Engine/Source/Runtime/Engine/Public/PrimitiveSceneProxy.h:295

它最重要的几个接口是：

  • DrawStaticElements
    Engine/Source/Runtime/Engine/Public/PrimitiveSceneProxy.h:444
  • GetDynamicMeshElements
    Engine/Source/Runtime/Engine/Public/PrimitiveSceneProxy.h:502
  • GetViewRelevance
    Engine/Source/Runtime/Engine/Public/PrimitiveSceneProxy.h:538
  • CreateRenderThreadResources
    Engine/Source/Runtime/Engine/Public/PrimitiveSceneProxy.h:586
  • DestroyRenderThreadResources
    Engine/Source/Runtime/Engine/Public/PrimitiveSceneProxy.h:593

这几个函数的组合，几乎完整定义了一个 primitive 对 Renderer 的“协作协议”：

GetViewRelevance

告诉 Renderer：

  • 我对当前 view 是否 relevant
  • 我应该走 static 还是 dynamic 路径
  • 我要不要进 main pass、depth pass、shadow pass、custom depth 等

DrawStaticElements

告诉 Renderer：

  • 如果我是静态相关对象，我能提供哪些可缓存的 mesh 元素

GetDynamicMeshElements

告诉 Renderer：

  • 如果我是动态相关对象，我在当前 view 下应该提交哪些 FMeshBatch

CreateRenderThreadResources / DestroyRenderThreadResources

告诉 Renderer：

  • 我在 RenderThread 侧还需要创建和销毁哪些专属渲染资源

这说明 FPrimitiveSceneProxy 并不是一个被动数据包，而是：

Renderer 对单个 primitive 发起渲染查询时的统一代表对象。

11. FPrimitiveSceneProxy 和 FPrimitiveSceneInfo 不要混

这两个名字很像，但职责不同。

FPrimitiveSceneProxy

偏“渲染代表对象”：

  • 保存线程安全的渲染数据
  • 暴露渲染接口
  • 决定 view relevance
  • 提供 mesh batches

FPrimitiveSceneInfo

偏“scene 内部管理对象”：

  • 被挂进 FScene
  • 参与 visibility、light interaction、draw command cache、GPUScene 等系统
  • 是 Renderer 内部管理 primitive 的核心壳对象

所以更容易记的说法是：

  • Proxy：我该怎么被渲染
  • SceneInfo：scene 里怎样管理我

12. 这套设计到底解决了什么问题

如果把这篇文章压成一句话，那就是：

UE 用 deferred render update + scene bridge + primitive proxy，把游戏线程对象和渲染线程对象拆开了。

这样做的收益非常明确：

  • RenderThread 不需要直接解引用 UObject
  • 组件变化可以按类型分类同步，而不是一律重建
  • scene 可以统一管理 primitive 的 add / update / remove
  • Renderer 可以通过一套稳定接口消费各种不同组件类型

这也是为什么：

  • GameThread 主要负责“准备和同步”
  • RenderThread 主要负责“组织和消费”

而不是两个线程一起去碰同一份对象状态。

结语

理解 GameThread、RenderThread、RHIThread 的协作，真正难的地方不是记住线程名字，而是搞清楚：

  • 哪些数据留在游戏对象里
  • 哪些数据会被镜像到渲染对象里
  • 哪些变化只是更新 transform
  • 哪些变化必须重建整个 render state

一旦把 UPrimitiveComponent -> FPrimitiveSceneProxy -> FPrimitiveSceneInfo 这条链理顺，后面再往下看：

  • FMeshBatch
  • MeshDrawCommand
  • SceneVisibility
  • GPUScene

就不会再把它们看成一堆分散的实现细节，而会看到它们其实都在消费同一套线程边界和渲染代理模型。

本文结论

  1. GameThread 负责准备这帧要画什么，RenderThread 负责组织怎么画，RHIThread 负责继续向底层图形 API 提交命令。
  2. UE 渲染线程协作本质上是流水线，而不是串行接力。
  3. 组件变化不会立刻跨线程同步，而是先通过 MarkRenderStateDirty、MarkRenderTransformDirty、MarkRenderDynamicDataDirty 打上分类标记。
  4. UWorld::SendAllEndOfFrameUpdates 是组件渲染变化在帧末统一结算的关键节点。
  5. UPrimitiveComponent::CreateRenderState_Concurrent 会通过 FScene::AddPrimitive 触发 CreateSceneProxy、创建 FPrimitiveSceneInfo 并排队 RenderThread 命令。
  6. 变换更新通常只走 UpdatePrimitiveTransform，不需要重建整个 proxy。
  7. MarkRenderStateDirty 的真实含义通常是 remove + add，也就是整套 render state 重建。
  8. FPrimitiveSceneProxy 是 UPrimitiveComponent 在渲染世界里的镜像对象，也是线程边界的核心。
  9. Renderer 最关心 FPrimitiveSceneProxy 的三个接口：GetViewRelevance、DrawStaticElements、GetDynamicMeshElements。
  10. FPrimitiveSceneInfo 偏 scene 管理，FPrimitiveSceneProxy 偏渲染代表，这两个概念不要混在一起。
