写在前面

前面的几篇文章，已经把 UE 主场景 Deferred Renderer 的主骨架基本串起来了：

  • 渲染准备阶段如何生成 ViewFamily、FSceneView 和 FViewInfo
  • RenderThread 上这一帧怎样从 PrePass 走到 PostProcessing
  • 后处理又怎样把 SceneColor 最终推进到 ViewFamilyOutput

但如果只停在这里，读源码时仍然会有一个非常强的疑问：

这些 pass 到底是怎样真正变成 GPU 工作的？

在 BasePass、Deferred Lighting、PostProcessing 这些函数里，看到的往往是：

  • GraphBuilder.AddPass(...)
  • FComputeShaderUtils::AddPass(...)
  • FPixelShaderUtils::AddFullscreenPass(...)
  • Pass->Draw(...)

而不是图形 API 教程里那种直白的：

  • DrawIndexedInstanced(...)
  • DispatchCompute(...)
  • vkCmdDrawIndexed(...)

这正是 UE 渲染源码第二层难点所在。
你已经能看懂“这一帧按什么顺序组织”，但还没真正打通：

这一帧的 C++ 渲染阶段，是怎样通过 RDG、RHI 和 Shader 最终落到 GPU 上的。

这篇文章就专门解决这个问题。

一条主线：从阶段组织到 GPU 执行

先把这篇文章最重要的主线放出来：

如果换一种更偏职责分层的说法，这条链也可以理解成：

这两张图就是本文的总纲。

1. 为什么前面已经看了很多 pass，却仍然会觉得“没看见 GPU”

第一次读 UE 渲染源码时，很容易产生一种奇怪的割裂感：

  • PrePass 看起来是在画东西
  • BasePass 看起来也在画东西
  • PostProcessing 更是一连串很明显的图像处理阶段

但在这些函数内部，真正直接像图形 API 那样“下命令”的地方并不显眼。
原因不是 UE 没有做这件事，而是它把这件事拆成了几层：

  1. 先由 renderer 决定“这一帧要有哪些阶段”
  2. 再由 RDG 把这些阶段组织成一张带资源依赖的图
  3. 每个 pass 执行时，再通过 FRHICommandList 录制具体的底层命令
  4. 最后由 RHI 后端把这些命令交给 D3D12 / Vulkan / Metal

也就是说，前面那些阶段函数更多是在做：

图组织与命令描述

而不是直接执行 GPU 硬件。

2. FSceneRenderBuilder：整帧执行的上层组织器

这条桥梁链里，最上面的一层是：

  • Engine/Source/Runtime/Renderer/Private/SceneRenderBuilder.cpp:1034
  • Engine/Source/Runtime/Renderer/Private/SceneRenderBuilder.cpp:1079

也就是：

  • FSceneRenderBuilder::AddRenderer(...)
  • FSceneRenderBuilder::Execute()

如果你把它放回主渲染链里看，它的位置非常清楚：

  • Engine 和 Renderer 已经把这帧的 FSceneRenderer 准备好了
  • SceneRenderBuilder 负责把这些 renderer 的工作挂到真正的执行轨道上

这里最关键的理解是：

FSceneRenderBuilder 自己并不实现某个具体渲染阶段，它更像是场景渲染的执行调度壳。

真正的重要动作，是在它往下调用的执行过程中触发：

  • Engine/Source/Runtime/Renderer/Private/SceneRenderBuilder.cpp:771
  • Engine/Source/Runtime/Renderer/Private/SceneRenderBuilder.cpp:915

也就是说，这一层真正做的是：

  • 把当前帧的 renderer 工作组织起来
  • 在合适的时机触发 RDG 图执行

3. FRDGBuilder：它不是命令列表，而是一张“渲染图”

RDG 的核心定义在：

  • Engine/Source/Runtime/RenderCore/Public/RenderGraphBuilder.h:47

而它最重要的说明，其实就在开头的注释里：

  • Engine/Source/Runtime/RenderCore/Public/RenderGraphBuilder.h:42
  • Engine/Source/Runtime/RenderCore/Public/RenderGraphBuilder.h:44

那里已经直接写明了：

  • Render Graph Builder 用来 build up a graph of passes
  • barrier 和 lifetime 来自 pass parameter struct 里的 RDG 参数
  • 图会在 Execute() 中被 compile、cull、execute

这几句非常关键，因为它把 RDG 和“普通命令数组”彻底区分开了。

RDG 不是：

  • 先把若干 draw / dispatch 存进一个数组
  • 然后顺序执行

RDG 更像是：

一张由 pass 节点和资源依赖关系组成的渲染图。

在这张图里，系统不仅知道“要做什么”，还知道：

  • 这个 pass 读什么
  • 写什么
  • 和其他 pass 有什么前后依赖
  • 哪些资源什么时候需要活着
  • 哪些 barrier 应该自动插入
  • 哪些 pass 可以被裁剪掉

这就是 RDG 相比传统手写 command list 最大的价值。

4. AddPass(...) 真正做的是什么

FRDGBuilder 最核心的接口是：

  • Engine/Source/Runtime/RenderCore/Public/RenderGraphBuilder.h:218
  • Engine/Source/Runtime/RenderCore/Public/RenderGraphBuilder.h:230
  • Engine/Source/Runtime/RenderCore/Public/RenderGraphBuilder.h:237

分别对应：

  • AddPass(...)
  • 无参数版本的 AddPass(...)
  • AddDispatchPass(...)

如果只看名字，最容易误解成：

调了 AddPass(...)，这个 pass 就已经在 GPU 上开始跑了。

但更准确的理解是：

AddPass(...) 是在给 RDG 图添加一个新的节点，并描述这个节点执行时要读写哪些资源、使用什么 workload、运行什么 lambda。

这里最重要的不是 lambda 本身，而是：

  • pass parameter struct
  • ERDGPassFlags
  • 资源字段上的 RDG 类型

因为是这些信息共同告诉 RDG：

  • 这个 pass 是 raster 还是 compute
  • 这个 pass 的资源依赖是什么
  • 它应该怎样被安排进整张图

所以从阅读角度看，AddPass(...) 最重要的作用不是“执行”，而是：

描述。

5. AddDispatchPass(...) 为什么名字容易让人误会

AddDispatchPass(...) 在：

  • Engine/Source/Runtime/RenderCore/Public/RenderGraphBuilder.h:237

如果第一次看到，很容易把它理解成：

“哦，这是 compute shader pass。”

其实不完全是这样。
它更准确地说，是一种让 pass 采用“显式发起命令录制任务”的组织方式。
在 UE 里，它既可以服务 compute，也可以服务某些并行 raster 路径。

这也是为什么你在 PrePass / BasePass 里会看到一种乍看有点奇怪的组合：

  • 用的是 AddDispatchPass(...)
  • 但 PassFlags 仍然是 ERDGPassFlags::Raster

所以这里“Dispatch”的重点不在 shader 类型，而在：

如何组织命令录制与任务分发。

6. Execute()：RDG 从“搭图”切换到“真正执行”的分界线

RDG 的执行入口在：

  • Engine/Source/Runtime/RenderCore/Public/RenderGraphBuilder.h:413

这一步在理解上非常重要。

在 Execute() 之前，代码大多在做：

  • CreateTexture(...)
  • CreateBuffer(...)
  • CreateSRV(...)
  • CreateUAV(...)
  • AllocParameters(...)
  • AddPass(...)

这些操作都还停留在：

搭图、描述依赖、描述执行方式

而一旦进入 Execute()，语义就变了：

  • 图会被编译
  • 图会被裁剪
  • pass 会按依赖和 workload 被真正安排执行
  • 对应 pass 的 lambda 会被真正跑起来

所以，如果要给整个 RDG 系统找一个最关键的分界点，那就是：

Execute() 之前是图构建，Execute() 之后才是图执行。

7. FRHICommandList：这才是“录制底层命令”的地方

更靠近底层的一层，是：

  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:3818

也就是 FRHICommandList。

它的关键作用，不是组织渲染图，而是：

承接某个 pass 在真正执行时需要录制的底层图形命令。

最典型的接口有：

  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:3973 DrawIndexedPrimitive(...)
  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:4259 BeginRenderPass(...)
  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:4291 EndRenderPass()
  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:2929 DispatchComputeShader(...)

这几个接口很关键，因为它们终于开始接近传统图形 API 的语义了。

也就是说：

  • RDG 解决的是“pass 之间怎么组织”
  • FRHICommandList 解决的是“某个 pass 具体要录什么命令”

8. ALLOC_COMMAND(...)：为什么说 UE 是在“录命令”，而不是“立刻执行”

看看 DrawIndexedPrimitive(...) 的实现：

  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:3973

最关键的是这两段：

  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:3978
  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:3981

逻辑很清楚：

  • 如果 Bypass()，直接走 RHIDrawIndexedPrimitive(...)
  • 否则走 ALLOC_COMMAND(FRHICommandDrawIndexedPrimitive)(...)

同样的结构也出现在：

  • BeginRenderPass(...)
  • EndRenderPass()
  • DispatchComputeShader(...)

对应位置分别是：

  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:4268
  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:4277
  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:4304
  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:4308
  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:2933
  • Engine/Source/Runtime/RHI/Public/RHICommandList.h:2936

这些代码说明：

大多数情况下，UE 不是在这里“直接让 GPU 执行”，而是在当前 RHI 命令流里分配并记录一条命令对象。

这也是为什么前面我们一直说：

  • CPU 在 RenderThread 上组织和录制
  • GPU 的真正执行发生在更靠后的底层提交阶段

9. 画图元和跑 Compute，在这里终于被统一起来了

从 FRHICommandList 这层往下看，Raster 和 Compute 其实有着完全一致的模式：

Raster

  • BeginRenderPass(...)
  • 绑定 PSO / shader / 参数
  • DrawIndexedPrimitive(...)
  • EndRenderPass()

Compute

  • 绑定 compute shader / 参数
  • DispatchComputeShader(...)

它们的差别主要在：

  • 使用什么类型的 pipeline
  • 使用什么类型的资源状态
  • 具体 shader 在 GPU 上如何运行

但从 CPU 侧的提交视角看，它们本质上都是：

把一条图形或计算命令记录进 RHI 命令流。

这点非常重要，因为它会让你以后再看：

  • BasePass
  • AddFullscreenPass
  • FComputeShaderUtils::AddPass

这些不同风格的代码时，不会再觉得它们是完全不同的世界。

10. Shader：前面的所有组织，最终都是为了把资源送进这里

如果说：

  • Renderer 决定“这一帧做什么”
  • RDG 决定“这些工作如何组织”
  • RHI 决定“命令怎样被录下来”

那么 shader 真正决定的，就是：

GPU 上每个像素、每个线程、每个图元到底做什么。

这一层最适合用 GenerateMips 来看：

  • Engine/Source/Runtime/RenderCore/Private/GenerateMips.cpp:15
  • Engine/Shaders/Private/ComputeGenerateMips.usf:1

在这个例子里，整个链条非常清楚：

  • FGlobalShader 定义 C++ 侧 shader class
  • BEGIN_SHADER_PARAMETER_STRUCT 定义资源和常量绑定契约
  • IMPLEMENT_GLOBAL_SHADER(...) 把它绑定到 .usf 里的入口函数
  • GraphBuilder.AllocParameters(...) 和 CreateSRV/CreateUAV(...) 准备资源
  • AddPass(...) / FComputeShaderUtils::AddPass(...) 把 shader 扔进 RDG pass
  • 最终通过 FRHICommandList 走向真正的 draw / dispatch

所以 shader 在这条桥梁链中的位置，可以准确理解成：

执行逻辑的终点。

前面所有 C++ 组织工作，最终都是为了把正确的资源、正确的参数、正确的入口函数送到这里。

11. 用 BasePass 重新回看这条桥梁链

现在可以用一个你已经熟悉的例子，把整条链再压一次。

以 BasePass 为例，它真正经历的是：

  1. FDeferredShadingSceneRenderer::Render 决定这一帧要进入 BasePass
  2. RenderBasePass(...) 组织：
    • render targets
    • depth / stencil 访问方式
    • shader 参数
    • mesh pass
  3. 通过 RDG 把它挂成一个 raster pass
  4. pass 真正执行时，通过 FRHICommandList 录制：
    • render pass begin/end
    • PSO 绑定
    • draw calls
  5. GPU 执行 BasePass shader，真正写出 SceneColor + GBuffer

也就是说，BasePass 在源码里既是：

  • 一个帧组织阶段

也是：

  • 一个 RDG 图节点

还是：

  • 一组 RHI 命令

最终才是：

  • GPU 上的一段真实 raster 工作

这四层叠在一起，才是它的完整身份。

12. 这条桥梁链对“自定义 Pass”意味着什么

如果你已经理解了这篇文章的主线，那么前面提到的“最小自定义 Pass”也就有了明确位置。

以一个插在 PostProcessing 里的最小全屏 Pass 为例，它在这条链中的位置就是：

这张图其实就是你以后自己往 UE 渲染主线里加东西时，最稳定的工作模板。

可以带走的结论

  1. 前面看到的 PrePass、BasePass、PostProcessing 等阶段函数，大多是在描述和组织 GPU 工作，而不是直接执行 GPU。
  2. FSceneRenderBuilder 负责把 scene renderer 的执行组织起来，并最终触发 RDG 图执行。
  3. FRDGBuilder 的本质是一张带资源依赖、生命周期和 barrier 信息的渲染图，而不是简单命令列表。
  4. AddPass(...) 的核心作用是声明一个 pass 节点，而不是立刻执行这个 pass。
  5. Execute() 是 RDG 从“图构建”切换到“图执行”的分界线。
  6. FRHICommandList 负责把某个 pass 的具体 draw / dispatch / render pass 操作录制成底层 RHI 命令。
  7. ALLOC_COMMAND(...) 说明 UE 更常见的模式是“录制命令”，而不是在每个调用点立刻直通 GPU。
  8. Shader 是这条桥梁链的执行终点：前面所有 C++ 组织工作，最终都是为了把资源和参数送进 .usf 的入口函数。
  9. 只要把“帧组织层、图组织层、命令录制层、GPU 程序层”这四层区分开，UE 渲染源码就会清晰很多。
  10. 自定义一个最小 Pass，本质上就是在这条桥梁链上插入一个新的图节点，并为它准备 shader、参数和输出。

写在最后

如果把前面的几篇文章和这一篇放在一起看，UE 主场景渲染最重要的主干其实已经基本打通了：

  • 从 ViewFamily、FSceneView 到 FViewInfo 的准备链
  • 从 RenderThread 入口到 PostProcessing 的整帧主线
  • 从 FSceneRenderBuilder、RDG、RHI 到 Shader 的提交桥梁

到这一步，再回头看 UE Renderer 目录，感受会和一开始很不一样。
它不再像是一堆相互独立的大文件，而会更像一套层次分明的系统：

  • 上层决定阶段顺序
  • 中层决定资源依赖
  • 底层决定命令录制
  • 最终由 shader 在 GPU 上执行

  • 继续沿现代渲染系统展开，分别看 Lumen、Nanite、Virtual Shadow Maps 是怎样插进主线的
  • 或者进入更偏实践的方向，专门写一篇“如何在 UE 渲染主线里添加一个最小自定义 Pass”
