系列定位：UE 渲染源码主线拆解
本文主题：从 ViewFamily 到 FViewInfo

写在前面

很多人第一次读 Unreal Engine 渲染源码，都会先去找这些文件：

  • DeferredShadingRenderer.cpp
  • BasePassRendering.cpp
  • PostProcessing.cpp

这当然没错，因为这些文件里确实有“真正的渲染流程”。
但问题也在这里。

如果一上来就直接扎进 pass 级代码，通常会遇到两个问题：

  1. 你能看见很多渲染阶段，却不清楚这些阶段到底依赖了哪些输入。
  2. 你能看见很多 View、ViewFamily、ViewInfo、SceneRenderer，却不知道它们是从哪里来的，又是在什么时候准备好的。

于是，整个 Renderer 看起来像一台已经运转起来的机器。你看得见齿轮在转，却不知道它是怎么启动的。

所以，本文不直接进入 PrePass、BasePass 和光照计算，而是先回答一个更基础的问题：

在真正开始渲染之前，UE 是如何把“这一帧谁在看、怎么看、画到哪、用什么规则画”整理成 Renderer 能消费的数据的？

把这件事看清楚，后面再读 Deferred Renderer，理解成本会低很多。

本文讨论的主题可以概括为：

视图描述的生成与 Renderer 交接

更通俗一点说，就是标题里的这句话：

UE引擎中的渲染准备工作

先给结论：本文到底在讲什么

如果只保留一句话，那么本文讨论的就是：

UE 不只是“执行渲染”，它还要先把一帧渲染需要的视图数据组织出来。

展开后可以分成 6 步：

  1. 游戏主视口每帧会从 UGameViewportClient::Draw 开始组织当前这一帧的场景绘制。
  2. UE 不会一上来就直接创建单个 View，而是先创建一个 FSceneViewFamily，作为一组视图共享的上下文。
  3. 然后 UE 会为每个本地玩家、每个眼睛或每个额外视角，构造出对应的 FSceneViewInitOptions。
  4. 接着 FSceneViewInitOptions 会被真正构造成 FSceneView，并补齐矩阵、裁剪、后处理和各种视图状态。
  5. 当 Engine 把这批 View 交给 Renderer 后，Renderer 会在 GameThread 上把 FSceneView 升级成 FViewInfo。
  6. 最后，经过 SceneRenderBuilder 的组织，这一帧才真正开始进入 RenderThread 和后续的 GPU 执行阶段。

也就是说，在 UE 中，一帧的开始并不是某个 pass 被执行，而是：

一个完整的视图系统被构建出来。

本文讨论范围

为了让主线尽量清晰，本文只聚焦下面这部分：

  • 运行时主游戏视口
  • 常规场景渲染主线
  • UGameViewportClient::Draw -> FSceneViewFamily -> FSceneView -> FViewInfo 这一条路径

不重点展开但会顺带提到的内容包括：

  • SceneCapture
  • 反射捕获
  • 编辑器视口
  • HitProxy 渲染
  • 后续 Deferred Renderer 的 pass 级执行细节

换句话说，本文主要回答的是：

在主场景一帧渲染开始前，CPU 侧到底完成了哪些准备工作。

从最上层开始：是谁在驱动这一帧

从主游戏视口的角度看，一帧绘制的驱动链大致如下：

  • UGameEngine::RedrawViewports
  • FViewport::Draw
  • UGameViewportClient::Draw

在当前源码里，对应位置大致是：

  • Engine/Source/Runtime/Engine/Private/GameEngine.cpp
  • Engine/Source/Runtime/Engine/Private/UnrealClient.cpp
  • Engine/Source/Runtime/Engine/Private/GameViewportClient.cpp

这条链回答了一个基础问题：

主游戏视口这一帧，是谁把“开始绘制”这件事真正触发起来的。

UGameEngine::RedrawViewports

这是上层调度入口。
在正常游戏运行时，Engine 会在合适的时机调用它，让当前可见的 viewport 开始这一帧的绘制。

可以把它理解成：

“这一帧该让各个视口重绘了。”

FViewport::Draw

进入 FViewport::Draw 后，UE 会创建 FCanvas，并把真正的绘制工作下发给 ViewportClient->Draw(...)。
这一层更像一个通用的“视口绘制外壳”。

UGameViewportClient::Draw

真正和主游戏场景渲染直接相关的工作，从这里开始变得具体。

因此，如果你研究的是：

  • 玩家主摄像机画面
  • 游戏场景是怎么开始一帧渲染的
  • Engine 到 Renderer 的第一跳

那么 UGameViewportClient::Draw 就是最自然的入口。

为什么先创建 ViewFamily，而不是先创建 View

这里先建立第一个核心认知。

在 UGameViewportClient::Draw 中，UE 不会先创建某个玩家相机的 FSceneView，而是先创建：

FSceneViewFamilyContext

这个对象背后的基类是 FSceneViewFamily。

FSceneViewFamily 是什么

它可以直接理解为：

一组视图共享的渲染上下文。

它内部最关键的东西包括：

  • 这组视图正在观察哪个 Scene
  • 最终画到哪个 RenderTarget
  • 这组视图共享哪些 EngineShowFlags
  • 这一组视图包含哪些 Views
  • 当前帧编号、时间信息、屏幕比例、扩展接口等

换句话说，FSceneViewFamily 不是某一个摄像机，而是这一批视图共享的壳。

为什么必须先有 family

原因很简单：一帧里不一定只有一个视图。

典型场景包括：

  • 普通单人游戏主相机：通常是 1 个 view
  • VR：左右眼至少是 2 个 view
  • 模拟分屏：多个玩家可能对应多个 view
  • 主视图之外的特殊视图：某些情况下也可能被纳入同一批构建逻辑

而这些 view 经常共享：

  • 同一个 scene
  • 同一个 render target 体系
  • 同一套 show flags
  • 同一帧的时间和屏幕比例设置

因此，从架构上看，先有 family，再往里面逐个加入具体 view，会比让每个 view 各自维护共享信息更合理，也更利于后续 Renderer 统一处理。

FSceneViewFamilyContext 的角色

FSceneViewFamilyContext 本身很薄。
它更像一个作用域对象，主要负责：

  • 继承 FSceneViewFamily
  • 在当前使用场景里管理 Views 的生命周期

它的重要性不在于逻辑复杂，而在于它明确表达了这件事：

这批 View 是临时组织起来的一组视图上下文。

FSceneViewFamily 到底装了什么

如果把 FSceneViewFamily 拆开看，大致可以分成四类信息。

1. 渲染目标和场景归属

这一类最基础：

  • RenderTarget
  • RenderTargetDepth
  • Scene

它们决定的是：

  • 这一组视图画到哪
  • 这一组视图看的是哪个场景

2. 共享渲染开关

最典型的是：

  • EngineShowFlags
  • ViewMode

它们决定的是：

  • 这组视图以什么模式在渲染
  • 哪些系统开启或关闭
  • 是否启用特定调试视图

3. 时序和帧信息

例如：

  • Time
  • FrameNumber
  • FrameCounter

这些信息后续会继续传到 Renderer 和 RenderThread，成为整帧统一的时间与帧级上下文。

4. 一组具体视图

最关键的成员之一是：

  • Views

也就是说，FSceneViewFamily 的设计就是“共享配置 + 具体子项”。

如果用一个简单图来概括：

在 ViewFamily 之上，UE 还会做哪些共享设置

在 UGameViewportClient::Draw 里，family 创建完成以后，UE 还会继续补一批 family 级别的配置。

这些设置不像 View 那样显眼，但它们决定了后续每个 view 工作在怎样的共同环境里。

典型内容包括：

  • ViewMode
  • EngineShowFlags 的覆盖和修正
  • Screen Percentage 相关设置
  • Dynamic Resolution
  • Stereo 渲染相关配置
  • HDR 标记
  • ViewExtensions

也就是说，在 single view 出现之前，UE 已经把“这组视图的共同渲染环境”处理了一遍。

从 ViewFamily 到单个 View：开始遍历玩家

当 family 级别的共享配置准备得差不多以后，UGameViewportClient::Draw 才会进入真正的“逐个 view 构造”阶段。

这里最核心的动作是：

  • 遍历所有本地玩家
  • 针对每个玩家决定要创建几个 view
  • 对每个 view 调用 ULocalPlayer::CalcSceneView

这一步有两个值得注意的点。

1. 一个本地玩家不一定只生成一个 view

最典型的情况就是 stereo。

如果当前是 VR 或立体渲染，那么单个玩家可能对应左右眼两个 view。
如果是某些分屏或模拟模式，一个玩家也可能拆成多个视图。

这再次说明：

ViewFamily 并不是一个“装一个 view 的容器”，而是一个真正的“视图集合上下文”。

2. 这里开始，UE 才真正进入 per-view 逻辑

family 之前处理的是共享问题。
从 CalcSceneViewInitOptions 和 CalcSceneView 开始，问题变成了：

  • 这个具体观察者的位置在哪里
  • 它的相机旋转是什么
  • 它的投影矩阵是什么
  • 它有哪些隐藏物体
  • 它有哪些后处理配置

也就是说：

这一步开始，UE 才真正从“这一帧的公共环境”走向“这个视角自己的描述”。

FSceneViewInitOptions：不是 View，本质上是“构造参数包”

在这条主线里，FSceneViewInitOptions 很关键，但也很容易被忽略。

很多人第一次读到它时，会把它和 FSceneView 混在一起。
其实两者职责完全不同。

FSceneViewInitOptions 负责什么

它负责：

在真正创建 FSceneView 之前，把构造一个 view 所需的原材料先装起来。

它不是最终对象，而是构造阶段使用的输入描述。

它为什么继承 FSceneViewProjectionData

因为 UE 希望在这个阶段就把最底层的投影几何数据准备好。

这部分内容包括：

  • ViewOrigin
  • ViewRotationMatrix
  • ProjectionMatrix
  • ViewRect
  • ConstrainedViewRect

这些数据还不代表一个完整 view，但已经足以构成“从世界到投影空间”的数学基础。

它比投影数据多出来什么

FSceneViewInitOptions 又在此基础上补充了更高层的语义：

  • ViewFamily
  • SceneViewStateInterface
  • ViewActor
  • PlayerIndex
  • StereoPass
  • ViewLocation
  • ViewRotation
  • HiddenPrimitives
  • ShowOnlyPrimitives
  • FOV
  • DesiredFOV
  • FirstPersonParams
  • bIsSceneCapture
  • bIsReflectionCapture
  • bIsPlanarReflection

一句话总结：

FSceneViewInitOptions 是“一个 view 应该如何被构造”的输入说明书。

CalcSceneViewInitOptions：把底层原材料先装箱

CalcSceneViewInitOptions 负责收集创建 view 所需的底层输入。最适合按来源理解。

第一类输入：投影与视口几何

首先，UE 会调用 GetProjectionData(...)，拿到：

  • 视口矩形
  • 裁剪后的矩形
  • 投影矩阵
  • 视图原点和旋转矩阵

这部分决定了这个 view 占据哪块屏幕区域，以及以什么投影方式观察世界。

第二类输入：相机附加状态

接下来会从 PlayerCameraManager 提取一些附加状态，例如：

  • fade color
  • color scale
  • 是否发生了 camera cut

这些数据不属于基础几何，但属于视图语义的一部分。

第三类输入：view state 与上下文

接着，UE 会确保当前 view 拥有可用的 SceneViewStateInterface。很多历史、时序和可见性相关的数据，都依赖这份持久化 state。

同时还会设置：

  • ViewActor
  • PlayerIndex
  • LODParameter
  • WorldToMetersScale
  • CursorPos
  • OriginOffsetThisFrame

这些数据共同决定了一件事：这个 view 不是一次性的相机快照，而是当前玩家、当前世界状态和当前时序条件下的正式视角。

CalcSceneView：从相机语义到正式 FSceneView

CalcSceneViewInitOptions 负责准备原材料，CalcSceneView 负责把这些原材料补齐成真正的视图语义，并构造成 FSceneView。

它先取的是 FMinimalViewInfo

CalcSceneView 里会调用 GetViewPoint(ViewInfo)，得到 FMinimalViewInfo。

这一步补齐了更接近相机语义的内容：

  • Location
  • Rotation
  • FOV
  • DesiredFOV
  • first person 参数
  • overscan / crop

这意味着 UE 在这里做的不是机械组装，而是把 Camera、PlayerController、ViewTarget 等高层概念压缩成渲染可用的数据。

它还会补 hidden / show only 逻辑

这里还会补上 hidden / show only 逻辑。view 并不是“看到整个 scene”，它还会带着：

  • HiddenPrimitives
  • ShowOnlyPrimitives

这一步决定了可见性处理阶段哪些物体被排除，哪些物体是唯一允许出现的。

然后才真正 new FSceneView

这里有一条很清晰的边界：在这之前，都是构造参数和语义准备；从这里开始，才真正得到一个正式的 FSceneView 对象。

FSceneView 到底是什么

可以用一句话概括 FSceneView：

FSceneView 不是相机本身，而是渲染器眼中的“一个观察者如何看世界”的正式描述。

1. 它不是 CameraActor

CameraActor、PlayerController、PlayerCameraManager、ViewTarget 都属于更高层的游戏语义。它们共同作用之后，才会在渲染层收敛成一个 FSceneView。

2. 它不是最终执行对象

FSceneView 很重要，但它仍然是 Engine 侧的描述对象。Renderer 真正执行时使用的是后面升级出来的 FViewInfo。

3. 它已经足够完整

虽然它还不是最终执行对象，但已经包含一套完整的视图信息：

  • 视口矩形
  • 视图矩阵
  • 投影矩阵
  • 相机位置和旋转
  • 过滤规则
  • capture / reflection / stereo 状态
  • 最终后处理配置

可以直接把 FSceneView 看成 Engine 侧的一份完整 per-view 渲染描述。

FViewMatrices：全文最值得反复看的部分

如果 FSceneView 是正式视图描述，那么 FViewMatrices 就是它的数学骨架。很多问题最后都会回到这里：

  • 这个矩阵是从哪里来的
  • 为什么这里有这么多矩阵
  • PreViewTranslation 到底是什么意思
  • 为什么投影矩阵会有“按 RHI 调整前后”的区别

这些问题大多都能在 FViewMatrices::Init 里找到答案。

一次性看懂 FViewMatrices

FViewMatrices 中最核心的字段可以简单整理成下面这张表。

字段 含义
ProjectionMatrix 当前 view 真正用于渲染的投影矩阵
InvProjectionMatrix 投影矩阵的逆
ViewMatrix 世界空间到 view 空间的变换
InvViewMatrix view 空间到世界空间的变换
ViewProjectionMatrix 世界到裁剪空间的完整变换
InvViewProjectionMatrix 裁剪空间反推回世界空间的完整变换
TranslatedViewMatrix 带 PreViewTranslation 的 view 变换
TranslatedViewProjectionMatrix 带 PreViewTranslation 的完整投影变换
ViewOrigin 当前视图在世界中的观察原点
PreViewTranslation 在真正进入投影前，对世界坐标做的平移

为什么会有这么多矩阵

因为渲染器面对的不是单一任务。它既要：

  • 把顶点从世界空间投到屏幕上

也要：

  • 从屏幕空间重建世界位置
  • 做反投影
  • 做可见性和裁剪
  • 计算各种 screen-space 效果
  • 处理大世界坐标精度问题

如果只有一份 ViewProjectionMatrix，很多事情都得在使用现场临时推导。UE 选择在 view 构建阶段就把高频矩阵体系一次性整理出来。

为什么这些矩阵在 CPU 侧准备

这里常见的疑问是：矩阵变换不是 GPU 最擅长的吗？为什么还要在 CPU 上算？

关键在于区分两件事：

  1. CPU 计算 per-view 常量矩阵
  2. GPU 用这些矩阵处理海量顶点和像素

这里 CPU 做的是第一件事。这些矩阵不是给每个顶点单独算的，而是每个 view 只准备一次，再作为 uniform buffer 提供给 GPU。

这样做很划算，原因有三个：

  • 这些矩阵是 per-view 的，不是 per-vertex 的，计算量很小。
  • CPU 自己在可见性、LOD、视图组织阶段也需要使用这些结果。
  • 先准备成稳定常量，再交给 GPU，大量后续 shader 都可以复用。

所以这里并不是“CPU 抢了 GPU 的工作”，而是 CPU 先把视图规则准备好，再交给 GPU 处理海量数据。

PreViewTranslation 为什么重要

PreViewTranslation 是 UE 里非常关键的概念。直觉上，世界坐标似乎应该直接参与变换，但 UE 会先把世界整体平移到当前 ViewOrigin 附近。原因很直接：这样可以显著改善浮点精度，尤其是在大世界坐标场景下。于是后续很多计算并不直接基于原始世界位置，而是基于“以当前 view 为中心重新平移过的世界空间”。

这也是为什么 FViewMatrices 不只有普通 ViewMatrix，还会有 TranslatedViewMatrix、TranslatedViewProjectionMatrix 等平移后的版本。

FSceneView 构造完成以后，事情还没结束

当 new FSceneView(ViewInitOptions) 发生时，view 的几何和视图空间描述已经齐了，但还缺一块关键内容：

最终画面风格。

这里说的“风格”不是艺术意义上的风格，而是：

  • Bloom
  • Exposure
  • Color grading
  • Motion blur
  • Lens flare
  • Tonemapping 相关配置
  • 各种 post process volume 混合后的最终设置

这些内容并不是在 FSceneView 构造函数里一次性写完的，而是在它加入 ViewFamily 之后，再走一轮专门的后处理配置收集流程。

为什么后处理配置不直接在构造函数里一次性写完

原因很简单：一个最终视图的后处理结果并不来自单一来源，而是多层输入的混合：

  • 引擎默认值
  • 项目默认值
  • 世界中的 PostProcessVolume
  • Camera 自己带的 post process 设置
  • Camera 动画和 modifier 的覆盖
  • Stereo 设备的额外修改

因此，UE 把后处理配置拆成了一个单独阶段：

StartFinalPostprocessSettings

这一步负责准备 FinalPostProcessSettings 的基线，包括：

  • 写入默认值
  • 应用项目和 CVar 的默认策略
  • 让 stereo 设备先做一轮初始化修正
  • 调用世界级别的 post process 设置收集逻辑

它回答的问题是：如果后面什么都不覆盖，这个 view 默认会长成什么样。

OverridePostProcessSettings

这一层负责真正的混合。它不是简单赋值，而是：

  • 只对被 override 的字段生效
  • 按权重对数值型参数做 lerp
  • 对少数枚举或引用类型参数做直接替换

也就是说，它不是替换整个结构体，而是对最终设置做字段级混合。

EndFinalPostprocessSettings

最后一步负责收尾和合法化处理：

  • 根据当前平台和 feature level 做最终修正
  • 根据 show flags / CVar 把某些项关掉或规整
  • 保证 FinalPostProcessSettings 成为一个“最终可执行版本”

后处理阶段的本质，就是把多个来源的 post process 语义合成为当前 view 的最终显示风格。

此时的 FSceneView 已经完整了吗

到这里可以说：对于 Engine 侧来说，FSceneView 已经基本完整了。

它现在已经同时具备了：

  • 空间和投影信息
  • 视图裁剪与过滤信息
  • stereo / capture / reflection 等状态
  • 完整的最终后处理配置

到了把 ViewFamily 交给 Renderer 的那一刻，Engine 侧交出去的不是半成品，而是一组已经准备好的视图描述。

Engine 到 Renderer 的正式交接点

在 UGameViewportClient::Draw 的后半段，所有玩家的 view 都准备好之后，Engine 会调用：

GetRendererModule().BeginRenderingViewFamily(SceneCanvas, &ViewFamily);

这是一个很清晰的边界。在这个调用之前，问题的主语一直是：

  • GameViewportClient
  • LocalPlayer
  • SceneView
  • ViewFamily

在这个调用之后，问题的主语开始变成：

  • RendererModule
  • SceneRenderer
  • ViewInfo
  • SceneRenderBuilder

这一步可以直接看成：

Engine 侧视图准备结束，Renderer 侧场景渲染构建开始。

BeginRenderingViewFamily：Renderer 接手后的第一跳，并不是立刻开画

BeginRenderingViewFamily 很容易让人误以为“Renderer 现在开始真正绘制了”。源码给出的答案是：还没有。

在这一步，Renderer 首先做的是：

  • 检查 ViewFamily 合法性
  • 确保所有 family 指向同一个 scene
  • 调用 World->SendAllEndOfFrameUpdates()，把 world 到 render proxy 的状态同步到最新
  • flush canvas
  • 设置 frame number
  • 整理 streaming view origins
  • 准备 upscaler / extension 相关的上下文

Renderer 接手后的第一阶段，更像是把“已经存在的视图描述”整理成“可以正式开始场景渲染构建”的状态。

也就是说，从架构上看：

FSceneRenderer：Renderer 的组织者，而不是某个 pass

在这条链里，FSceneRenderer 是一个关键类型。FSceneView 描述的是单个观察者如何看世界，FSceneRenderer 则更像是 Renderer 为这一组 view 创建出来的整帧组织者。

它的构造函数里有一句非常值得记住的检查：

check(IsInGameThread());

这句话明确说明：即使已经进入 Renderer，FSceneRenderer 的构造和准备工作仍然发生在 GameThread。这也再次说明，本文讨论的核心仍然是渲染准备，而不是 GPU 上真正的绘制执行。

从 FSceneView 到 FViewInfo：为什么 Renderer 不直接用前者

当 FSceneRenderer 被构造时，它会遍历 ViewFamily->Views，然后把每个 FSceneView 复制成 FViewInfo。

这一步说明，Renderer 不直接以 FSceneView 作为最终执行对象。

为什么要多一层 FViewInfo

FSceneView 虽然已经是一份完整的视图描述，但它更偏向语义层描述。Renderer 真正执行时，还需要一整套额外的运行时状态。

这些状态包括：

  • primitive visibility map
  • primitive relevance map
  • static mesh visibility map
  • 动态 primitive 收集器
  • occlusion query 相关状态
  • renderer-side 的 uniform cache
  • fog / water / translucency 等后续 pass 会用到的额外字段

这些正是 FViewInfo 要补上的内容。

它本质上是一轮职责升级

可以把这次变化理解成：

所以，FViewInfo 不是一个“不同的 view”，而是 Renderer 为了真正执行渲染，对 FSceneView 做的一次执行语义升级。

一句话概括两者关系

  • FSceneView：Engine 侧的正式视图描述
  • FViewInfo：Renderer 侧的正式执行视图

FViewInfo 到底比 FSceneView 多了什么

如果把这件事说得再具体一点，可以直接用表格来对照。

维度 FSceneView FViewInfo
归属 Engine 侧 Renderer 侧
核心职责 描述一个视角如何看世界 承载这个视角后续真正渲染执行所需的状态
是否有矩阵和视图语义 有 有，且继承自 FSceneView
是否有可见性和 occlusion 状态 很少 大量具备
是否直接作为 Renderer 的主要工作单元 否 是

最重要的区别可以压缩成一句话：FSceneView 负责“看”，FViewInfo 负责“干活”。

到这里为止，RenderThread 其实还没真正开始“执行渲染”

即使 ViewFamily 已经交给 Renderer，即使 FSceneRenderer 和 FViewInfo 也已经构造出来了，RenderThread 依旧还没有开始执行真正的帧内 pass。真正把工作推向 RenderThread 的，是后面的：

  • FSceneRenderBuilder
  • AddRenderer(...)
  • Execute()

也就是说，本文讨论的整个系统，仍然主要处在：

构建、整理、封装、交接

而不是：

执行 PrePass、BasePass、Lighting、PostProcess

可以用一句话概括：

本文讨论的是“如何把一帧装好”，而不是“如何把一帧画出来”。

最值得带走的 8 个结论

最后，把全文压成 8 个最值得记住的结论。

1. 一帧主场景渲染在 Engine 侧的重要入口是 UGameViewportClient::Draw

如果你想研究主游戏视口这一帧是怎么开始组织的，从这里入手最自然。

2. UE 先创建 ViewFamily，再创建具体 View

这说明 UE 先解决“一组视图共享什么”，再解决“单个视图长什么样”。

3. FSceneViewInitOptions 不是最终视图对象，而是构造参数包

它负责在正式创建 FSceneView 之前，先把投影、相机、过滤规则和状态接口等原材料装起来。

4. FSceneView 是 Engine 侧正式的视图描述

它已经包含：

  • 视图矩阵
  • 投影矩阵
  • view rect
  • capture / stereo 等状态
  • 最终后处理配置

5. FViewMatrices 是视图数学骨架

它负责整理 ViewMatrix、ProjectionMatrix、ViewProjectionMatrix 以及 PreViewTranslation 等关键变换关系。

6. 后处理配置是在 FSceneView 加入 family 之后再灌进去的

它不是简单复制，而是初始化、混合和收尾修正的三段式流程。

7. Renderer 接手后，第一阶段仍然主要发生在 GameThread

BeginRenderingViewFamily 之后，先做同步和整理，再构建 FSceneRenderer。

8. Renderer 真正使用的是 FViewInfo，不是原始的 FSceneView

FViewInfo 是在 FSceneView 基础上增加 Renderer 执行状态后的正式工作对象。

结语：为什么这部分值得先看

最容易产生的误解是：“这不就是渲染前的一点准备工作吗？是不是不如直接去看真正的 pass？”

恰恰相反。正因为后续所有 pass 都建立在这些 view 数据之上，所以这部分不是边角料，而是 Renderer 的地基。

如果没有先理解：

  • FSceneViewFamily
  • FSceneViewInitOptions
  • FSceneView
  • FViewInfo

之间的关系，那么后面看到的 Deferred Renderer 往往只会是一堆复杂函数调用。你知道它在做事，却很难真正知道它为什么能在这里做这件事。

一旦这条链被看清楚，后面的 PrePass、BasePass、光照与后处理就都有了一个共同前提：它们都是围绕已经准备好的 view 在展开。

如果把全文最重要的收获压缩成一句话，就是：

在 UE 中，一帧渲染真正开始之前，最重要的工作不是某个 pass 被执行，而是整套视图描述被构建并完成交接。
