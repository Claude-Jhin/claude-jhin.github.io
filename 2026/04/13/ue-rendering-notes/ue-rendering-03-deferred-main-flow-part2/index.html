本文主题：从 AddPostProcessingPasses 到 ViewFamilyOutput

写在前面

上一篇文章把主场景 Deferred Renderer 的前半段主线串了起来，从 RenderViewFamily_RenderThread 一直到 Fog / Cloud / Water / Translucency。
到那一步为止，一帧画面在视觉上已经很接近完整结果了，但它还没有真正走完“输出”这条链。

这一篇只做一件事：

把 UE 的 PostProcessing 读成一条真正的输出流水线。

很多人在看后处理时，容易把它理解成一组附加特效：TAA、Bloom、Tonemap、FXAA、Debug Overlay、Upscale。
但如果直接看源码，会发现 UE 的后处理并不是“最后加几个效果”，而是：

以 SceneColor 为核心工作资源，持续消费深度、速度、透明分离结果、曝光和 Bloom 等中间资源，最终把这一帧写入 ViewFamilyOutput 的输出链。

对应的源码入口有两个：

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3943
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:347

前者负责把控制权交给后处理，后者才是后处理主线真正展开的地方。

1. 后处理从哪里接手

在 DeferredShadingRenderer 的后半段，Renderer 会对每个 View 调用：

  • Engine/Source/Runtime/Renderer/Private/DeferredShadingRenderer.cpp:3943

这一步很重要，因为它说明：

  • 前面的几何、光照、环境、透明阶段，到这里都已经基本结束
  • 后处理接手的对象，不再是 mesh 或 GBuffer 的建立过程
  • 接手的是一张已经经历过大部分渲染阶段的场景结果

也就是说，PostProcessing 在整帧中的位置，不是“某个孤立特效模块”，而是整帧最后的统一接管点。

2. 一进入后处理，哪些资源已经在手里

AddPostProcessingPasses(...) 的定义在：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:347

函数开头最关键的几行是：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:381
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:382
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:383
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:385
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:388

也就是：

  • ViewFamilyOutput
  • ViewFamilyDepthOutput
  • SceneDepth
  • Velocity
  • PostDOFTranslucencyResources

这几项资源非常有代表性。它们说明后处理从一开始就同时依赖：

  • 场景颜色输出目标
  • 场景深度
  • 运动向量
  • 分离透明结果
  • 当前 view 的输出语义

如果只看功能名，后处理容易显得很杂；但从资源角度看，它其实非常统一：

它接手了一组已经由前面主线建立好的关键资源，然后围绕这些资源不断做重建、合成、转换和输出。

3. SceneColor 是这条链里的核心工作资源

理解后处理最重要的入口，不是某个具体 pass，而是这句注释：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:395

这句基本就是 PostProcessing.cpp 的总纲。

围绕这句注释，代码里又明确建立了几个关键变量：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:399 SceneColorBeforeTonemapSlice
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:400 SceneColorAfterTonemap
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:403 OriginalSceneColor

这说明在后处理阶段，SceneColor 至少有 4 种重要语义：

  • 当前正在被不断改写的工作纹理
  • 刚进入后处理时的原始场景颜色
  • Tonemap 之前的 HDR 场景颜色
  • Tonemap 之后的输出域颜色

所以后处理真正的阅读姿势不是“这个 pass 做了什么特效”，而是：

当前这一步把 SceneColor 从什么状态变成了什么状态。

4. PassSequence：后处理不是一堆 if，而是一条显式输出链

后处理最值得注意的组织结构，是：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:546

后面会继续做：

  • SetNames(...)
  • SetEnabled(...)
  • Finalize()

相关位置包括：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:528
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:546
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:727

这说明 UE 并不是把后处理硬编码成一条固定死的链，而是先：

  1. 定义一组标准后处理阶段
  2. 根据 View、ShowFlags、AA 模式、Debug 模式、ViewExtension、Post Process Material Chain 等条件决定哪些阶段启用
  3. 再把启用的阶段串成一条有效输出链

所以 PostProcessing 的工程结构，更像是：

一个以 ViewFamilyOutput 为终点的可裁剪、可覆盖、可插拔输出序列。

这也是为什么它能同时容纳：

  • TAA / TSR
  • Motion Blur
  • Tonemap
  • FXAA / SMAA
  • 各种 Visualization
  • Editor Primitive 叠加
  • Upscale
  • AlphaInvert

5. 为什么 TAA / TSR 会在后处理里

很多人第一次看 UE 后处理，会对这一点最困惑：

为什么 TAA / TSR 不在前面的主渲染阶段里，而是在后处理链里？

关键位置在：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:977
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:984

分别对应：

  • AddMainTemporalSuperResolutionPasses(...)
  • AddGen4MainTemporalAAPasses(...)

这两类 pass 的输入不是 mesh，不是 GBuffer 建立过程，而是：

  • 当前帧的 SceneColor
  • SceneDepth
  • SceneVelocity
  • 历史帧数据
  • 分离透明结果

从本质上说，TAA / TSR 处理的是：

如何在屏幕空间和时间维度上重建更稳定的最终图像。

这已经完全属于输出链问题，而不是几何或光照建立问题。

所以它们在后处理里，不是例外，反而很自然。

6. SeparateTranslucency 是怎么并回主颜色的

在后处理中，还有一条很容易被忽略的资源支线：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:388 PostDOFTranslucencyResources

后面围绕它的关键逻辑在：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:831
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:870
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:908

这一段说明：

  • 前面有些透明结果并不是直接写进主场景颜色
  • 它们可能先被单独保存
  • 到后处理链的某个合适时机，再决定怎样 compose 回 SceneColor

之所以要这样做，是因为它们会和：

  • DOF
  • TSR
  • Material Chain
  • Post-DOF / Post-MotionBlur 的透明路径

这些步骤发生耦合。

所以后处理链还承担了另一项职责：

把前面为正确性而暂时分离保存的图像结果，在合适的时机重新并回主颜色。

7. Motion Blur 仍然是在消费 SceneColor / Depth / Velocity

Motion Blur 的主调用位置在：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1091

它吃进去的关键输入包括：

  • SceneColor
  • SceneDepth
  • SceneVelocity
  • 某些情况下的 VelocityFlattenTextures

这说明 Motion Blur 虽然在用户视角下是一个“效果”，但在源码结构上，它本质上是：

基于已有场景颜色和运动信息，对当前输出图像做时序相关模糊处理的输出阶段。

它不是独立于主链之外的额外修饰，而是主输出链中的一环。

8. 曝光、Histogram、Local Exposure、Bloom 为什么都在 Tonemap 前

这一段是后处理最像“资源生产链”的部分。

关键位置包括：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1230 AddHistogramPass(...)
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1264 AddBasicEyeAdaptationPass(...)
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1275 AddHistogramEyeAdaptationPass(...)
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1330 AddFFTBloomPass(...)
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1388 AddGaussianBloomPasses(...)

这几段共同说明：

  • 后处理不是只改一张现成的图
  • 它会先对 SceneColor 降采样
  • 再生成 Histogram、EyeAdaptation、LocalExposure、Bloom 等中间资源
  • 最后把这些资源喂给 Tonemap

所以这里的逻辑更像：

这些东西必须在 Tonemap 前，是因为它们处理的仍然是 HDR 场景结果，而不是最终输出图像。

9. Tonemap 是整条后处理链里最重要的分界点

Tonemap 主调用在：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1473

而在这之前，代码已经明确保存了：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1401 SceneColorBeforeTonemapSlice

Tonemap 完成后，又显式记录：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1483 SceneColorAfterTonemap = SceneColor

这说明 Tonemap 不是普通一环，而是整条链的语义分界线：

  • Tonemap 前，SceneColor 更接近 HDR 场景结果
  • Tonemap 后，SceneColor 更接近显示设备和最终输出图像

这也是为什么 AfterTonemapping 的材质链会同时读到：

  • PreTonemapHDRColor
  • PostTonemapHDRColor

对应位置在：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1515
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1516

10. FXAA / SMAA、可视化与调试阶段，为什么也放在后处理里

Tonemap 之后，后处理并没有结束。

接下来还会继续处理：

  • FXAA / SMAA
  • AfterTonemapping 的 Post Process Material Chain
  • VisualizeHDR
  • VisualizeMotionVectors
  • PixelInspector
  • SelectionOutline
  • EditorPrimitive
  • VirtualShadowMap / Nanite / GBuffer 等各类可视化

相关位置包括：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1483 之后的整段
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1503
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1828
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1866

这说明 PostProcessing 在工程上的职责不只是“做最终图像美化”，它还是：

  • 最终显示叠加层
  • 调试可视化层
  • 编辑器复合层

也就是说，凡是应该在图像即将输出前统一处理的东西，最后都会汇入这里。

11. ViewFamilyOutput：这条链从一开始就知道自己的终点

后处理一开头就创建了：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:381 ViewFamilyOutput

后面的 PassSequence 也是围绕它建立的：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:546

真正能看出“整条链就是以它为终点”的，是后面的几条检查：

  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:1970
  • Engine/Source/Runtime/Renderer/Private/PostProcess/PostProcessing.cpp:2003

这几行非常有代表性。它们说明：

无论中间经历多少 pass、多少中间纹理、多少 slice 和 override，最后 SceneColor 都必须回到 ViewFamilyOutput。

从工程设计上看，这就是“输出链”最明确的定义。

12. 把整个资源流收束成一张图

如果把后处理最关键的资源流压成一张图，大概就是这样：

这张图最重要的不是细节，而是它把后处理的本质说清楚了：

后处理并不是一堆互不相干的效果，而是一条以 SceneColor 为核心工作资源、最终导向 ViewFamilyOutput 的输出流水线。

可以带走的结论

  1. PostProcessing 在 UE 里不是附加模块，而是整帧真正的最终输出链。
  2. AddPostProcessingPasses(...) 接手的对象不是 mesh，而是已经经过光照、环境和透明合成后的场景结果。
  3. SceneColor 是这条链里的核心工作资源，会在多个阶段被持续改写。
  4. SceneDepth、Velocity、SeparateTranslucency 不是边角输入，而是时序重建、景深、运动模糊和透明合成的核心依赖。
  5. PassSequence 说明后处理在工程上是一条显式组织、可覆盖、可裁剪的输出序列。
  6. TAA / TSR 之所以属于后处理，是因为它们处理的是图像重建和历史帧稳定，而不是几何绘制。
  7. Histogram、EyeAdaptation、Local Exposure、Bloom 是 Tonemap 前的重要输入链。
  8. Tonemap 是整条输出链最关键的语义分界点。
  9. FXAA / SMAA、可视化、编辑器叠加和 Upscale 都属于“图像即将输出前的统一接管层”。
  10. ViewFamilyOutput 是整条后处理链明确的最终落点。

写在最后

如果把前一篇和这一篇合在一起看，UE 主场景 Deferred Renderer 的大骨架就已经基本完整了：

  • 前半段负责建立深度、GBuffer 和受光后的场景表达
  • 后半段负责环境、透明与体积合成
  • 最后由后处理链把整帧结果转换成真正的输出图像

到这一步，再回头看 FDeferredShadingSceneRenderer::Render(...)，它就不再像一大段难以下口的 if / pass / feature 分支，而会更像一条组织严密的整帧调度链。

如果继续沿这条系列展开，比较自然的两个方向是：

  • 回到更底层，专门拆 RDG / RenderCore / RHI 是怎样把这些 pass 组织和提交到 GPU 上的
  • 继续沿主渲染骨架往外扩，分别看 Lumen、Nanite、Virtual Shadow Maps 是怎样插入这条主线的
