学完着色器之后，我很自然地遇到了下一个问题：

如果 GPU 已经能根据顶点数据和 Shader 画出一个带颜色渐变的三角形，那么我们要怎样把一张真正的图片贴到图形上？

这就是 LearnOpenGL 里“纹理”这一节要解决的问题。

对应的参考页面是：LearnOpenGL CN - 纹理

这一节我在本地做了一个更贴近原教程的版本，项目路径是：

  • projects/learnopengl/learnopengl/05_textures

这次我不再只是在代码里生成一张最小纹理，而是把教程里的两张图片真正接进了 OpenGL：

  • container.jpg
  • awesomeface.png

当这个矩形最终被画出来时，我第一次比较完整地把这条链路串起来了：

  • 图片文件如何读进内存
  • 图片数据如何上传到 GPU
  • 纹理坐标如何和顶点绑定
  • sampler2D 如何在片元着色器里完成采样
  • 两张纹理又是怎样通过 mix 混到一起的

如果说上一节“着色器”让我真正开始理解 CPU 和 GPU 的配合方式，那么这一节则让我开始理解：

“屏幕上的细节，究竟是怎样从一张磁盘上的图片一路走到片元颜色里的。”

为什么纹理这一节很关键

只靠顶点颜色，能做的事情其实很有限。

你可以画出红绿蓝渐变的三角形，也可以通过 uniform 做一些动态变化，但如果想表现真实物体的表面细节，只靠顶点颜色远远不够。

因为顶点数量通常很少，而表面细节通常很多。

这时候，纹理就成了最自然的答案：

  • 把颜色信息预先存进一张图片
  • 再在渲染时按坐标去采样这张图片
  • 最终把采样结果作为片元颜色的一部分

从这个角度看，纹理并不是“给物体贴图这么简单”。

它本质上是：

“把大量表面信息从顶点数据里解耦出来，改成通过采样读取。”

这也是为什么后面继续学下去时，纹理不只会表示颜色，还会表示：

  • 法线
  • 高度
  • 镜面反射信息
  • 阴影信息
  • 环境信息

所以这节看起来只是“给矩形贴两张图”，实际上它是在为后面很多高级主题打基础。

这一节到底做了什么

这次本地项目里，我主要做了 6 件事：

  1. 用 EBO 画了一个矩形，而不是继续只画单个三角形
  2. 给每个顶点加上了纹理坐标属性
  3. 使用 stb_image 从磁盘加载真实图片
  4. 创建了两个 OpenGL 纹理对象
  5. 把两张纹理分别绑定到纹理单元 0 和 1
  6. 在片元着色器里通过 mix 混合采样结果

这 6 件事连起来，刚好就是“纹理这一节”的骨架。

从三角形变成矩形：为什么这次多了 EBO

这一节里，图形不再是一个三角形，而是一个矩形。

矩形在 OpenGL 里通常会拆成两个三角形来画，所以我这次的顶点数据变成了 4 个顶点，索引数据则告诉 GPU 应该怎样把它们组装成两个三角形。

也就是说：

  • 顶点数组负责提供“有哪些点”
  • 索引数组负责提供“这些点如何组成三角形”

这样做的好处很直接：

矩形的两个三角形可以复用顶点，而不是把重复顶点再写一遍。

这也是我第一次更明确地感受到 EBO 的职责：

它并不存放顶点属性本身，而是存放“顶点之间的引用关系”。

顶点数据为什么又多了一组纹理坐标

在这一节里，每个顶点包含三类信息：

  • 位置
  • 颜色
  • 纹理坐标

其中最重要的新成员就是纹理坐标。

纹理坐标通常记作 u, v，也就是二维坐标。它们不是顶点在屏幕上的位置，而是：

“当前顶点对应纹理图片上的哪个位置。”

例如一个矩形的四个角，通常会对应到图片的四个角：

  • 左下角对应 (0.0, 0.0)
  • 右下角对应 (1.0, 0.0)
  • 左上角对应 (0.0, 1.0)
  • 右上角对应 (1.0, 1.0)

这样，当光栅化阶段在矩形内部生成片元时，GPU 就会自动插值这些纹理坐标，最终为每个片元算出一个对应的采样位置。

也就是说，纹理坐标和前一节的颜色传递，其实在机制上很像：

  • 都是从顶点出发
  • 都会经过插值
  • 最后在片元着色器里被使用

只不过这一次，插值后的不是颜色，而是“去图片上取样的位置”。

glVertexAttribPointer(...) 在这一节又进了一步

这一节里，顶点布局变成了：

  • 3 个 float 表示位置
  • 3 个 float 表示颜色
  • 2 个 float 表示纹理坐标

所以每个顶点一共是 8 个 float。

也正因为如此，glVertexAttribPointer(...) 里的 stride 和 offset 也跟着变了。

它的含义是：

  • location = 2
  • 每个顶点读取 2 个 float
  • 每个顶点的总步长是 8 个 float
  • 从第 7 个和第 8 个 float 开始读，也就是纹理坐标段

这又一次提醒我：

glVertexAttribPointer(...) 不是某种机械背诵的模板，而是在精确描述“VBO 里的数据该怎样拆出来给 Shader 用”。

顶点着色器在这一节里做了什么

这一节的顶点着色器反而比上一节更简单了。

它的职责主要只有两件事：

  1. 把顶点位置写给 gl_Position
  2. 把纹理坐标传给片元着色器

也就是说，顶点着色器这次不再专注“产生一个有趣的视觉效果”，而是更像一个数据中转站：

  • 顶点位置照常参与顶点处理
  • 纹理坐标继续向后传

这种变化也让我更清楚地理解了一个事实：

顶点着色器不一定总是在“做复杂计算”，很多时候它只是负责把合适的数据传到后面的阶段。

为什么还需要 stb_image

学到这里时，我有一个很直接的问题：

OpenGL 既然能创建纹理对象，为什么不能直接“读图片文件”？

答案是：OpenGL 负责的是图形 API，它并不负责解析 JPG、PNG 这类图片格式。

所以如果想把磁盘上的图片变成 GPU 能用的像素数据，就需要一个额外的图片加载库。

这次我使用的是 stb_image。

它的作用可以概括成一句话：

“把 JPG、PNG 这些图片文件解析成一块连续的像素内存。”

当 stbi_load(...) 返回成功后，我拿到的是：

  • 图片宽度
  • 图片高度
  • 通道数
  • 指向像素数据的指针

而这些信息，正是 glTexImage2D(...) 所需要的输入。

所以 stb_image 解决的是“文件格式解析”问题，OpenGL 解决的是“把像素数据交给 GPU”问题。

两者职责并不重叠，但刚好接在一起。

第一次看到这句时，我最直接的感受是：“为什么图片还要翻转？”

后来理解下来，其实是在处理坐标系差异。

多数图片文件会把左上角视为原点，而 OpenGL 纹理坐标默认把左下角视为 (0, 0)。

如果不处理这个差异，图片通常就会出现上下颠倒的问题。

所以这句代码并不是某种“经验魔法”，它本质上是在做一次坐标系对齐。

这也是我在这一节里第一次明显感受到：

图形学里很多看起来奇怪的细节，其实背后都是不同坐标系统之间的转换问题。

创建纹理对象时，OpenGL 真正在做什么

这一节里，我真正开始把“纹理对象”理解成一个完整状态对象，而不只是一张图片。

创建纹理时，流程大致是这样的：

  1. glGenTextures(...) 创建纹理对象 ID
  2. glBindTexture(GL_TEXTURE_2D, texture) 绑定这个纹理对象
  3. glTexParameteri(...) 设置采样与边界行为
  4. glTexImage2D(...) 上传图片像素
  5. glGenerateMipmap(...) 生成 mipmap

这里最值得注意的是：

纹理对象不只是“像素数据本身”，它还携带很多和采样相关的状态，比如：

  • 边界如何处理
  • 缩小时如何过滤
  • 放大时如何过滤

也就是说，OpenGL 里的纹理对象更像是：

“图片数据 + 采样规则”的组合。

GL_TEXTURE_WRAP_* 和 GL_TEXTURE_MIN/MAG_FILTER 到底在控制什么

这一节里我第一次认真去理解这几个参数，而不是只把它们当作教程里固定要抄的几行代码。

纹理环绕方式

它们控制的是：

如果纹理坐标超出了 [0, 1] 范围，OpenGL 应该怎样处理。

GL_REPEAT 的含义是重复纹理。

这在当前这个例子里不算特别明显，但一旦把纹理坐标改得超过 1.0，就能立刻看到平铺效果。

纹理过滤方式

它们控制的是：

当纹理被缩小或放大时，采样结果应该如何插值。

GL_LINEAR 会让结果更平滑，而不是像像素块一样生硬。

这也让我第一次更明确地意识到：

“采样图片”并不是简单地取某一个像素，它背后还牵涉到如何从离散像素中估计连续结果的问题。

为什么 container.jpg 用 GL_RGB，awesomeface.png 用 GL_RGBA

这次加载两张图片时，我特意把两者分开理解：

  • container.jpg 没有透明通道，所以使用 RGB
  • awesomeface.png 带透明通道，所以使用 RGBA

这个区别非常重要，因为它直接决定了 glTexImage2D(...) 里像素格式该怎样写。

如果格式和图片真实通道不匹配，轻则显示效果不对，重则会出现更难排查的问题。

这一点让我对“通道数”这个概念第一次有了更具体的感受：

图片文件格式并不只是扩展名不同，它还影响 GPU 应该怎样理解上传进去的数据。

这一页真正打通了哪条链路

如果把这次的完整流程压缩成一条主线，我会这样理解：

  • 顶点数据提供位置和纹理坐标
  • VAO/VBO/EBO 负责把这些数据组织好并送进渲染管线
  • stb_image 把 JPG/PNG 解析成像素数据
  • OpenGL 纹理对象把这些像素数据上传到 GPU
  • 纹理单元把多个纹理对象组织成可供 Shader 访问的输入
  • 顶点着色器把纹理坐标传下去
  • 片元着色器通过 sampler2D 采样并混合两张纹理
  • 最终把结果写到屏幕上

如果把它写成一条更稳定的文字流程，大概是这样：

这一节最容易混淆的几个点

1. 纹理对象和图片文件不是一回事

  • 图片文件是磁盘上的资源
  • 纹理对象是 GPU 侧可被采样的数据对象

2. 纹理对象和纹理单元不是一回事

  • 纹理对象持有数据
  • 纹理单元是 Shader 采样器读取纹理时用的插槽

3. 纹理坐标不是屏幕坐标

纹理坐标描述的是“去纹理图片的哪里取样”，不是“顶点在窗口里的位置”。

4. stb_image 和 OpenGL 的职责不同

  • stb_image 负责读懂图片格式
  • OpenGL 负责把像素数据变成可渲染的纹理

结语

如果“着色器”这一节让我第一次理解了 GPU 程序怎样决定顶点位置和片元颜色，那么“纹理”这一节则让我第一次真正理解：

图片数据是怎样参与渲染的。

它不再只是“把一张图贴到矩形上”这么简单，而是让我开始看清：

顶点坐标、纹理坐标、采样器、纹理对象和片元颜色，究竟是怎样一步一步连到一起的。
