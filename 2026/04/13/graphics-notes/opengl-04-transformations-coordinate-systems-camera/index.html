学到 LearnOpenGL 的入门部分中段时，我明显感觉到内容开始“连起来”了。

前面的窗口、三角形、着色器、纹理，更多是在回答这些问题：

  • OpenGL 程序怎么跑起来
  • 顶点数据怎么送进 GPU
  • Shader 怎么决定颜色
  • 图片怎么参与渲染

而从“变换”开始，问题突然换了一个层次：

  • 一个物体为什么会出现在那个位置
  • 为什么它会旋转、缩放、远近变化
  • 相机移动时，场景为什么看起来会跟着变
  • 三维点最终又是怎样落到二维屏幕上的

这一篇笔记把三个章节合在一起记录：

  • 变换
  • 坐标系统
  • 摄像机

对应的本地项目分别是：

  • projects/learnopengl/learnopengl/06_transformations
  • projects/learnopengl/learnopengl/07_coordinate_systems
  • projects/learnopengl/learnopengl/08_camera

之所以把这三节放在一起，是因为它们本来就在回答同一件事：

“场景中的物体和相机，究竟是怎样通过矩阵被组织起来的。”

如果说前面的入门章节是在搭一个渲染程序，那么从这里开始，OpenGL 才真正进入“空间”的部分。

为什么这三节应该放在一起看

如果单看“变换”，你会学到：

  • 如何让一个物体平移、旋转、缩放
  • 如何把这些操作写成矩阵
  • 如何把矩阵传进顶点着色器

如果单看“坐标系统”，你会学到：

  • 一个顶点为什么不止属于一个空间
  • model / view / projection 三个矩阵怎样分工
  • 为什么远处物体会看起来更小

如果单看“摄像机”，你会学到：

  • view 矩阵不一定手写死
  • 它可以由相机位置和朝向实时生成
  • 鼠标、键盘、滚轮怎样控制观察方式

但把这三节连起来看，逻辑会清楚得多：

  1. 先学会让单个物体动起来
  2. 再学会把物体放进一个完整的三维空间链路里
  3. 最后再学会从“固定观察”升级到“可交互摄像机”

也就是说，这三节其实共同完成了一次认知升级：

从“我在画一个东西”，到“我在组织一个三维场景”。

第一部分：变换到底解决了什么问题

刚开始学 OpenGL 的时候，顶点坐标通常是写死的。

如果一个三角形的三个顶点写在某几个位置，它就只能老老实实待在那里。想让它挪一下位置，最直接的办法似乎就是改顶点数组。

但这种方法很快就会变得笨重：

  • 平移一次，要改所有顶点
  • 旋转一次，要重新算所有坐标
  • 缩放一次，还是要动所有数据

这时候，矩阵的价值就体现出来了。

矩阵的意义不在于“它长得像一张 4x4 表”，而在于：

“它把对顶点的批量几何操作，统一变成了一次规则明确的数学变换。”

这一步让我第一次真正理解：

物体的位置变化，并不是靠“改顶点数组”实现的，而是靠“用矩阵重新解释这些顶点”。

顶点本身没变，变的是它们通过什么坐标变换进入屏幕。

GLM 为什么在这里很重要

这一阶段我也正式把 GLM 接进了工程。

它的存在非常合理，因为如果没有一个专门的数学库，这几节会被大量矩阵细节拖得很重。

GLM 最直接的价值有三个：

  1. 它提供了和图形学高度贴合的类型
  2. 它把平移、旋转、缩放这些常见操作都封装好了
  3. 它和 OpenGL 的矩阵传递方式天然契合

例如：

  • glm::mat4
  • glm::translate(...)
  • glm::rotate(...)
  • glm::perspective(...)
  • glm::lookAt(...)
  • glm::value_ptr(...)

这让我可以把注意力放在“我想做什么变换”，而不是把时间全花在手写矩阵元素上。

这也是下一节“坐标系统”会进一步放大的主题。

第二部分：为什么一个顶点会经历这么多空间

到了 07_coordinate_systems，我开始真正接触这样一条变换链：

local -> world -> view -> projection

也就是：

  • 局部空间
  • 世界空间
  • 观察空间
  • 投影/裁剪相关空间

这一步最大的变化，是我不再只看“一个矩阵能不能让物体动起来”，而是开始问：

“一个顶点从原始定义到最终显示，中间到底经历了哪些坐标系统？”

这时候，Shader 里最核心的一句就变成了：

model、view、projection 到底各自管什么

这三个矩阵刚开始看起来都像“把顶点乘一下”，很容易混在一起。

但一旦把它们和空间概念对起来，分工就很清楚了。

model

model 决定的是：

“这个物体在世界里摆在哪里。”

也就是把物体从自己的局部空间，放到整个场景的世界空间里。

在 07_coordinate_systems 里，我用一组位置数组来放 10 个立方体：

  • 有的在前面
  • 有的更远
  • 有的偏左偏右

每个立方体都是同一个标准立方体，但通过不同的 model 矩阵，它们被放到了世界里的不同位置。

view

view 决定的是：

“相机怎么看这个世界。”

在坐标系统这一节里，我最开始是通过把整个场景向后平移，来模拟相机站在前面看。

也就是说，view 矩阵负责把世界空间转换成观察空间。

projection

projection 决定的是：

“怎样把三维点投到二维屏幕上。”

这里最重要的是透视投影。

这一步并不是在“移动物体”，而是在改变三维空间怎样被映射到屏幕。

为什么这一节开始让我真正有了“三维”的感觉

在前面的变换里，虽然已经有矩阵，但对象本质上还是一个平面矩形。

而到了坐标系统这节，我第一次在同一个画面里看到：

  • 多个立方体
  • 不同的深度位置
  • 透视投影带来的远近变化
  • 深度测试带来的遮挡关系

也就是从这一刻开始，我对“这是一个三维场景”有了更真实的感受。

因为三维不只是多了一个 z 坐标，而是多了：

  • 世界中的摆放关系
  • 相机视角
  • 投影方式
  • 深度遮挡

这几件事合在一起，空间感才真正成立。

第三部分：摄像机为什么是这三节的自然终点

如果说前两节是在学习“空间如何被组织”，那么 08_camera 做的事情，就是把“观察者”也正式加入这个系统。

这一步的关键变化不是 model，也不是 projection，而是 view。

而这正是“摄像机”这个章节真正的本质。

glm::lookAt(...) 到底在做什么

lookAt 的三个参数其实非常直观：

  • 相机在哪：gCameraPos
  • 相机朝哪看：gCameraPos + gCameraFront
  • 哪边算上：gCameraUp

也就是说，它是在根据相机的位置和朝向，自动构造观察矩阵。

这里我最大的收获是：

相机本身并不是某种“特殊对象”，它本质上就是一组状态：

  • 位置
  • 朝向
  • 上方向

只要这组状态确定了，view 矩阵就可以被算出来。

键盘、鼠标、滚轮分别在控制什么

摄像机这一节的交互让我把“相机状态”理解得更具体了。

键盘

W/A/S/D 控制的是相机位置。

也就是说：

  • W/S 沿前方向前后移动
  • A/D 沿右方向左右移动

这里我第一次比较明确地看到：

“左右”并不是固定的世界坐标方向，而是由当前视角动态算出来的。

鼠标

鼠标控制的是视角方向。

代码里用 yaw 和 pitch 来表示水平和垂直转角，再通过三角函数把它们转成 gCameraFront。

这一步非常关键，因为它把“鼠标移动”变成了一个真正的方向向量更新过程。

滚轮

滚轮控制的是 fov，也就是视野角。

它并不是把相机位置往前推，而是通过投影矩阵改变“看到的范围”。

所以滚轮带来的那种“拉近 / 拉远”的感觉，本质上是投影变化，而不是位移变化。

deltaTime 为什么是摄像机里最容易被低估的细节

在摄像机这一节里，还有一个特别重要但容易被忽略的点：

它解决的是帧率相关的问题。

如果不乘 deltaTime，相机每一帧都会走固定距离，那么：

  • 高帧率机器移动会更快
  • 低帧率机器移动会更慢

而乘上 deltaTime 之后，速度就变成了“每秒走多少”，而不是“每帧走多少”。

这虽然不是坐标系统本身，但却是摄像机能不能正常工作的一个非常现实的工程细节。

到这里，三条线终于合并了

如果把这三节的内容压缩成一条主线，我现在会这样理解：

  1. model 负责把物体从局部空间放进世界
  2. view 负责把世界转换成相机看到的样子
  3. projection 负责把三维投到二维屏幕
  4. camera 则让 view 从静态矩阵变成可交互生成的矩阵

也就是说：

  • “变换”让我理解了矩阵怎样作用于物体
  • “坐标系统”让我理解了顶点会经过哪些空间
  • “摄像机”让我理解了观察者本身如何进入这个系统

这时候再回头看 Shader 里的那一句：

它就不再只是一条需要背下来的模板，而是已经能拆出清楚的含义：

  • 先决定物体在世界里在哪里
  • 再决定相机怎么看它
  • 再决定如何把它投到屏幕上

这三节最容易混淆的几个点

1. model、view、projection 都是矩阵，但职责完全不同

  • model：物体在哪
  • view：相机怎么看
  • projection：怎么投到屏幕

2. 相机移动，本质上不是“相机对象自己真的被画出来了”

更准确地说，是观察矩阵根据相机状态重新生成了。

3. 滚轮缩放不等于相机前进

当前例子里，滚轮改的是 fov，属于投影变化，不是位置变化。

4. 真正的三维场景必须考虑深度测试

否则前后遮挡关系会错乱。

到目前为止，我确认自己已经掌握的东西

把这三节放在一起看，到这里我希望自己至少已经真正掌握了这些点：

  1. 知道为什么图形学里需要矩阵来做几何变换
  2. 知道 model / view / projection 三个矩阵各自负责什么
  3. 知道一个顶点为什么会经历多个空间
  4. 知道透视投影为什么会让远处物体更小
  5. 知道 glm::lookAt(...) 是怎样生成观察矩阵的
  6. 知道摄像机的移动、转向和缩放分别影响什么

结语

如果前面的章节让我学会了“把东西画出来”，那么这三节真正让我开始理解了：

图形学里的“空间”到底是怎么被组织出来的。

物体不是随便摆在屏幕上的，相机也不是某个抽象概念。它们都通过矩阵和坐标系统，被纳入到同一条严格而统一的变换链路里。

而我现在也越来越能感觉到，后面的很多内容，无论是光照、模型加载还是更复杂的场景渲染，其实都会继续站在这三节打下的基础上往前走。
