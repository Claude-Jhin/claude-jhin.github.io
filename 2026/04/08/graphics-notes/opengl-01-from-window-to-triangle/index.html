刚开始学 OpenGL 时，我以为“画出第一个三角形”只是一个象征性的入门仪式。真正动手之后才发现，这一步的意义比想象中大得多。

因为从“打开一个窗口”到“屏幕上出现一个三角形”，中间实际上串起了图形程序最基础的一整条链路：窗口系统、OpenGL 上下文、函数加载、Shader、顶点数据、GPU 缓冲区，以及最终的绘制调用。

这篇文章记录的，就是我刚开始学习 OpenGL 时，如何从一个空窗口走到第一个三角形，以及在这个过程中建立起来的最初理解。

当前的学习主线是 LearnOpenGL CN，开发环境选择的是 Visual Studio + GLFW + GLAD + OpenGL 3.3 Core。

目前为止，我完成了两件事：

  1. 跑通第一个 OpenGL 窗口
  2. 在窗口里成功绘制出第一个三角形

看起来只是很小的一步，但它已经把 OpenGL 最基本的运行方式完整打通了。

为什么要把“窗口”和“三角形”认真学一遍

很多教程在这一部分看上去都很简单，代码量也不大，于是很容易让人产生一种错觉：

“这几章只是铺垫，真正重要的是后面的纹理、光照和模型加载。”

但现在回头看，我反而觉得最初这两步特别值得慢一点学。因为后面的几乎所有内容，都是在这里建立起来的：

  • 为什么图形程序需要渲染循环
  • 为什么必须先创建 OpenGL 上下文，再初始化 GLAD
  • Shader 在整个渲染流程里到底扮演什么角色
  • 顶点数据为什么不能直接交给 GPU，而是要通过 VBO 和 VAO 这类对象来组织
  • 真正触发绘制的，到底是哪一句代码

如果这些问题一开始没有想明白，后面即使“能照着教程做出来”，理解也会非常飘。

GLFW、GLAD 和 OpenGL 到底各自负责什么

这是我入门时第一个需要理清的问题。

OpenGL：真正负责渲染的图形 API

OpenGL 本质上是一套图形编程接口。它定义了程序如何和 GPU 沟通，去完成这些事情：

  • 创建缓冲区
  • 上传顶点数据
  • 编译和使用 Shader
  • 配置渲染状态
  • 发出绘制命令

一句话概括：OpenGL 负责“怎么画”。

GLFW：负责窗口、上下文和事件

GLFW 不是图形 API，它更像是 OpenGL 程序和操作系统之间的桥梁。

它负责的事情包括：

  • 创建窗口
  • 创建 OpenGL 上下文
  • 处理键盘和鼠标输入
  • 处理窗口事件
  • 交换前后缓冲

一句话概括：GLFW 负责“在哪画，以及怎么和系统打交道”。

GLAD：负责加载 OpenGL 函数

GLAD 是 OpenGL 函数加载器。

现代 OpenGL 的很多函数，不是在程序启动时就天然可以直接调用的。必须先创建好 OpenGL 上下文，再通过驱动去查询这些函数的地址。

GLAD 做的，就是把这些函数入口加载进来，让程序真正可以使用它们。

一句话概括：GLAD 负责“把 OpenGL 函数接进程序”。

三者之间的典型顺序

实际执行顺序通常是这样的：

  1. 用 GLFW 创建窗口
  2. 用 GLFW 创建 OpenGL 上下文
  3. 用 GLAD 加载 OpenGL 函数
  4. 然后才真正开始调用 OpenGL 进行渲染

这条顺序非常重要，尤其是第 2 步和第 3 步不能反过来。

4. glClearColor 和 glClear 的区别

这是我刚开始最容易混淆的点之一。

  • glClearColor(...)：设置清屏颜色
  • glClear(...)：真正执行清屏

也就是说，前者是在设置状态，后者是在执行动作。

这其实也让我第一次意识到，OpenGL 里有很多 API 都是在“设置当前状态”，真正生效往往要等到后面的某个操作触发。

顶点着色器和片元着色器到底在做什么

顶点着色器：决定顶点被放到哪里

顶点着色器处理的是“每一个顶点”。

在当前这个最小例子里，它的工作非常简单：

  • 接收一个顶点位置 aPos
  • 把它写入 gl_Position

也就是说，它决定的是：

“这个顶点最终会出现在屏幕空间里的什么位置。”

虽然现在只是简单地原样传递位置，但这已经是后面变换、摄像机、投影矩阵等内容的起点了。

什么是 VAO

VAO 的全称是 Vertex Array Object。

它不是用来存顶点值本身的，而是用来记录：

“这些顶点数据应该怎么被解释。”

比如：

  • 顶点属性位置 0 从哪里读
  • 每个顶点包含几个分量
  • 每个分量是什么类型
  • 相邻顶点之间步长是多少

如果用一个更形象的比喻：

  • VBO 是数据仓库
  • VAO 是数据说明书

VBO 告诉 GPU“数据放在哪”，VAO 告诉 GPU“应该怎么读这些数据”。

到目前为止，我对这条渲染链路的理解

如果用一句话把“第一个三角形”背后的流程串起来，我现在的理解是：

  • CPU 先准备顶点数据
  • 通过 VBO 把数据上传到 GPU
  • 通过 VAO 告诉 OpenGL 如何解释这些数据
  • 通过顶点着色器处理每个顶点的位置
  • 通过片元着色器决定像素颜色
  • 最后通过 glDrawArrays 发出绘制命令

目前最容易混淆的几个概念

GLFW、GLAD、OpenGL

  • GLFW：负责窗口、上下文、输入和事件
  • GLAD：负责加载 OpenGL 函数
  • OpenGL：负责真正的图形渲染

顶点着色器和片元着色器

  • 顶点着色器：决定顶点放到哪里
  • 片元着色器：决定像素是什么颜色

VBO 和 VAO

  • VBO：存数据
  • VAO：存“怎么读这些数据”的规则

glClearColor 和 glClear

  • glClearColor：设置清屏颜色
  • glClear：真正执行清屏

这一路踩到的两个很真实的问题

1. Visual Studio 原生项目会不断重复配置库路径

如果每章都新建一个 .vcxproj，就会不断重复配置：

  • Include 路径
  • Library 路径
  • glfw3.lib
  • opengl32.lib

这类问题虽然不属于图形学本身，但会非常消耗学习精力。

后来我把这些内容抽到了公共 .props 文件里，整个工程组织才稳定下来。

2. 中文注释带来的源码编码问题

在 Visual Studio + CL.exe 这套环境下，如果源码编码和编译器读取编码不一致，中文注释有时会直接把代码解析搞乱，最后报出一堆看起来很奇怪的语法错误。

最后的解决办法是：

  • 源码统一保存为 UTF-8
  • 编译选项里统一加上 /utf-8

这也是为什么我后来把 /utf-8 也写进了公共属性表。

这个问题虽然和 OpenGL 没有直接关系，但它让我更直观地意识到：学习图形学，不只是学图形 API，还要学会和真实的开发环境打交道。

到现在为止，我真正学到的是什么

如果只用一句话概括，那么到现在为止，我学到的还不是“我会画一个三角形”，而是：

“我开始知道 OpenGL 程序到底是怎样一步一步运行起来的。”

更具体一点，我已经建立起了下面这些非常基础、但也非常关键的认识：

  1. OpenGL 程序不是一次性执行逻辑，而是持续运行的渲染循环
  2. 窗口系统、OpenGL 上下文、函数加载和图形 API 本身是不同层次的事情
  3. Shader 是运行在 GPU 端的小程序
  4. 顶点数据必须通过缓冲区和属性配置，GPU 才知道该怎么使用它们
  5. 真正触发绘制的是 draw call，而不是前面那些准备动作

这些东西现在看起来都还很初级，但它们正是后面继续往纹理、变换、摄像机、光照推进时最重要的地基。

下一步准备做什么

接下来我会继续沿着 LearnOpenGL 往后推进，重点想放在这几个方向：

  1. 继续做“你好，三角形”这一章的练习
  2. 更深入地理解 Shader 的编译、链接和使用方式
  3. 开始接触顶点属性扩展，例如颜色插值
  4. 为后面的纹理、变换和坐标系统打基础

结语

从空窗口到第一个三角形，这一步看起来很小，但它真正让我第一次把这些原本零散的词串到了一条完整链路里：

  • 窗口系统
  • OpenGL 上下文
  • 函数加载
  • Shader
  • 顶点数据
  • GPU 缓冲区
  • 绘制命令

而我现在也越来越能感受到，后面那些看起来更“炫”的内容，比如纹理、坐标变换、摄像机、光照，本质上都是在这条最基础的链路上继续叠加出来的。

所以现阶段最重要的事情，并不是急着往更高级的效果冲，而是把这条基础链路真正理解透。
